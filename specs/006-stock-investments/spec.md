### User Story 1 - 開啟股票頁立即掌握目前持倉與報酬率（Priority: P1）

使用者點開「股票」頁，最上方一張投資組合總覽卡顯示總市值、總成本、總損益（含 ▲／▼）、整體報酬率四個關鍵數字；下方排列每檔持股的個股卡片，每張卡顯示股票代號、名稱、持有股數、平均成本、目前股價、市值、損益、報酬率，獲利綠色（▲）虧損紅色（▼）。整頁不需任何點擊就能回答「我目前帳上賺多少／賠多少」。

**Why this priority**: 持倉總覽是工具核心價值的載體 — 使用者從多個券商 App 同時看部位是常態，本功能要成為「一個頁面看全部」的單一可信來源。如果連目前持倉都看不到，後面的交易紀錄、股利、實現損益都失去意義。屬 P1，且為其他所有 user story 的資料底座。

**Independent Test**: 一位有 5 檔持股、每檔皆有買入紀錄的使用者開啟股票頁；應立即看見：(1) 投資組合總覽卡顯示四個彙整數字（總市值、總成本、總損益、報酬率），(2) 5 張個股卡片，每張卡顯示代號／名稱／股數／平均成本／目前股價／市值／損益／報酬率八個欄位，(3) 損益正值綠色 ▲、負值紅色 ▼，(4) 持有股數為 0 的歷史代號**不**顯示於總覽（僅出現於交易紀錄頁）。任何資訊缺漏視為失敗。

**Acceptance Scenarios**:

1. **Given** 使用者持有 3 檔股票（2330、0050、6488），**When** 開啟股票頁，**Then** 總覽卡顯示三檔持股的總市值（Σ 持股數 × 目前股價）、總成本（Σ FIFO 剩餘成本）、總損益（總市值 − 總成本）與報酬率（損益 ÷ 成本 × 100%）。
2. **Given** 使用者賣光某檔持股至股數 = 0，**When** 開啟股票頁，**Then** 該檔**不**出現於個股卡片列表，但仍可在「實現損益」Tab 與「交易紀錄」Tab 看見其賣出歷史。
3. **Given** 某檔持股目前股價尚未取得（new add 後尚未查價），**When** 觀看個股卡片，**Then** 市值欄位以「—」呈現、損益與報酬率亦為「—」，不阻擋其他持股的數值顯示。
4. **Given** 使用者持股總體未實現損益為 +1,200 / 報酬率 +5.3%，**When** 觀看總覽卡，**Then** 損益數字以綠色配 ▲ 符號顯示；若為 −800 / −2.1% 則紅色配 ▼。

### User Story 2 - 紀錄一筆股票買賣並即時看到費用試算（Priority: P1）

使用者點「新增交易」開啟 Modal，選擇買入／賣出（按鈕切換時顏色即時變化：買入綠、賣出紅），輸入股票代號（系統自動從 TWSE 查名稱與股價並回填）、日期、股數（強制整數）、每股價格、扣款／入款帳戶、備註；手續費與證交稅依政府公告稅率自動計算（手續費 0.1425%／整股最低 20 元／零股最低 1 元；證交稅僅賣出時依股票類型 0.3%（一般）／0.1%（ETF / 權證）計算，最低 1 元）。Modal 下方即時顯示費用摘要（成交金額、手續費、交易稅、總成本／淨收入）讓使用者送出前看到確切金額。

**Why this priority**: 紀錄交易是本功能最高頻的操作；任何讓使用者「手動算手續費／稅再輸入」的設計都會把人逼回 Excel。即時試算 + 自動帶入是使用者願意持續記錄的關鍵。屬 P1。

**Independent Test**: 使用者輸入「股票 = 2330、股數 = 1000、每股 = 600」並選「買入」；費用摘要應即時顯示「成交金額 600,000 / 手續費 855 / 交易稅 0 / 總成本 600,855」；切到「賣出」後手續費與成交金額不變但交易稅顯示 1,800（600,000 × 0.3%）、淨收入顯示 597,345。送出後該筆紀錄出現於交易紀錄頁，且該帳戶餘額同步扣款／入款（買入扣 600,855、賣出入 597,345）。

**Acceptance Scenarios**:

1. **Given** 使用者開啟新增交易 Modal、選擇「買入」，**When** 輸入代號 2330 並等待 500ms，**Then** 系統自動查 TWSE 並在輸入欄旁顯示「✓ 台積電 即時 $600（13:25）」綠色提示，同時自動填入名稱（若該檔尚未建立持倉則送出時一併新增該檔股票）。
2. **Given** 使用者輸入無效代號 「9999」，**When** 等待 500ms，**Then** 輸入欄旁顯示「找不到此股票代號」紅色提示；送出按鈕仍可按下（系統不阻擋），但送出時若仍無法查到名稱則以「（未命名）」建立持倉。
3. **Given** 使用者輸入股數 = 1.5（小數），**When** 試圖送出，**Then** 系統拒絕並顯示「股數必須為整數」；不送出 request。
4. **Given** 使用者選擇 ETF（如 0050）並輸入「賣出 1000 股 × $130」，**When** 觀看費用摘要，**Then** 證交稅 = ⌊130,000 × 0.1%⌋ = 130 元（非一般股的 0.3%）；手續費仍為 ⌊130,000 × 0.1425%⌋ = 185 元。
5. **Given** 使用者輸入「買入 1 股 × $50」（零股），**When** 觀看費用摘要，**Then** 手續費為 1 元（零股最低，非整股的 20 元）。
6. **Given** 使用者切換買入／賣出按鈕，**When** 切換完成，**Then** 按鈕配色即時變化（買入綠、賣出紅），費用摘要區塊同步重算（賣出多顯示證交稅、總額顯示淨收入而非總成本）。

### User Story 3 - 記錄股利並支援自動同步除權息（Priority: P1）

使用者點「新增股利」開啟 Modal，輸入除息日期（ex-dividend date，法定基準日，非實際入帳日）、股票代號（同樣 TWSE 自動查名稱）、現金股利或股票股利至少填一項（股票股利會增加持有股數）、入款帳戶、備註。或者點股利紀錄頁的「同步除權息」按鈕，系統自動以 TWSE 公開資料補齊歷史與本年度的除權息資料（依除息日當下的持股數計算）。

**Why this priority**: 股利是台股投資的重要報酬來源（殖利率動輒 4–6%），缺漏會讓持倉報酬率嚴重失真。手動記錄繁瑣；自動同步省下使用者手動翻券商 App 的時間。屬 P1，與交易紀錄並列為記帳兩大支柱。

**Independent Test**: 使用者新增一筆股利「2330 / 除息日 2025-09-12 / 現金股利 4 / 股票股利 0 / 持股 1000」，送出後現金股利欄位自動算為 4,000、股票股利為 0；觀看股利紀錄頁可看到該筆。接著按「同步除權息」按鈕，系統依該使用者最早交易日期到今日掃 TWSE 除權息列表，補齊所有「該使用者除息日當下持有 ≥ 1 股」的歷史股利紀錄；同日同股票若已存在則不重複新增；自動新增的紀錄備註標示「TWSE自動同步（每股 $X.XX）」。

**Acceptance Scenarios**:

1. **Given** 使用者點開新增股利 Modal、輸入代號 0050，**When** 等待 500ms，**Then** 系統自動查 TWSE 並回填名稱「元大台灣 50」與目前股價（價格僅供參考，不寫入股利金額）。
2. **Given** 使用者除息日 2025-09-12 持有 2330 共 1000 股、輸入現金股利每股 4，**When** 送出，**Then** 系統自動計算現金股利金額 = 1000 × 4 = 4,000；股利寫入帳戶後該帳戶餘額立即增加 4,000（與既有交易記帳一致）。
3. **Given** 使用者於股利紀錄頁按「同步除權息」按鈕，且其最早交易日為 2024-01-15、今日 2026-04-26，**When** 系統執行同步，**Then** 系統依年份分段查 TWSE TWT49U（2024、2025、2026）、僅處理該使用者已持有的代號、依除息日當下持股數計算每筆股利金額並寫入；同日同股票若資料庫已有股利紀錄則跳過不重複；備註自動標示「TWSE自動同步（每股$X.XX）」。
4. **Given** 含股票股利的個案（如 2330 配 0.5 元現金 + 每千股配 100 股股票股利），**When** 同步完成，**Then** 系統呼叫 TWT49UDetail 取得每股現金與每千股配股數；現金股利金額 = 持股數 × 0.5（四捨五入）、股票股利股數 = 持股數 × 100 ÷ 1000，且增加該檔的持有股數。
5. **Given** 使用者於除息日 2025-09-12 完全無持股（同日先前已賣光），**When** 同步執行，**Then** 該檔該日的股利**不**列入處理（持股 = 0 跳過）。

### User Story 4 - 設定定期定額並由系統自動於登入時產生交易（Priority: P2）

使用者點「定期定額」Tab，新增一筆定期定額排程：選擇股票、設定每期預算、週期（每週／每月／每季）、起始日、扣款證券帳戶、備註、啟用／停用。系統在使用者每次登入時檢查排程，依「每期預算 ÷ 當前股價」計算可買股數（無法買 1 股時略過）並自動產生買進交易。

**Why this priority**: 定期定額是長期投資者的核心習慣，但需要紀律與自動化才能持續。本功能將「下單」這一段交給使用者本人於券商 App 完成，但「記錄」交給系統自動，補齊「我這個月有沒有按計畫扣款」的提醒功能。屬 P2，因為核心查看與紀錄能力（US1 ~ US3）已可獨立交付，本層在其上疊加「不主動操作也持續累積資料」。

**Independent Test**: 使用者建立一筆「2330 / 每月 / 5,000 / 起始日 2026-01-01 / 啟用」的定期定額。每次登入系統檢查：若距上次產生 ≥ 1 個月且當前股價可整除買 1 股以上，則自動產生一筆買入交易（如當前股價 600 → 可買 8 股 / 4,800、剩 200 不入帳）。若排程日恰逢週末或 TWSE 休市日，自動順延至下一個交易日，交易紀錄日期寫入實際交易日，備註附「原排程 YYYY-MM-DD 順延」；排程節奏不變（下次仍依原起始日推算）。

**Acceptance Scenarios**:

1. **Given** 使用者建立定期定額「0050 / 每月 / 6,000 / 起始日 2026-02-01 / 啟用」，**When** 使用者於 2026-02-02 登入（系統當前 0050 股價 130），**Then** 系統自動產生一筆買入交易：日期 2026-02-02 / 股數 = ⌊6000 ÷ 130⌋ = 46 股 / 每股 130 / 帳戶 = 設定的扣款帳戶；該帳戶餘額同步扣款（含手續費）。
2. **Given** 排程日 2026-03-01 恰逢週日（TWSE 休市），**When** 使用者於 2026-03-02 (週一) 登入，**Then** 系統將實際交易日寫為 2026-03-02、備註「原排程 2026-03-01 順延」；下次排程仍以原起始日推算為 2026-04-01（節奏不變）。
3. **Given** 排程「2330 / 每月 / 1,000」但當前股價 600，**When** 排程觸發，**Then** 系統計算 ⌊1000 ÷ 600⌋ = 1 股、總成本 600 + 手續費 20（整股最低）= 620；若預算僅 500 則跳過該次（未滿 1 股）並記錄略過原因供使用者查看。
4. **Given** 使用者將某筆排程改為「停用」，**When** 下次登入，**Then** 系統**不**為該排程產生任何交易；若再次改回「啟用」則僅從下次自然觸發點開始（不補產生停用期間漏掉的交易）。
5. **Given** TWSE 休市日資料尚未快取，**When** 系統執行排程檢查，**Then** 系統呼叫 TWSE `/v1/holidaySchedule/holidaySchedule` API、過濾掉「開始交易／最後交易」等特別交易日後快取 24 小時，避免每次登入都打 API。

### User Story 5 - 檢視 FIFO 實現損益並了解每筆賣出的真實獲利（Priority: P2）

使用者點「實現損益」Tab；頂部彙總卡顯示總實現損益、整體報酬率、今年實現損益、已實現筆數四個數字；下方表格列出每筆賣出交易的日期、股票、股數、賣出均價、成本均價（FIFO）、手續費 + 稅、實現損益、報酬率。FIFO 邏輯保證：每筆賣出對應到「最早買入的那批」的成本基礎，完全一致於券商實際稅務認定方式。

**Why this priority**: 賣出後的「真實已實現獲利」是評估投資績效的客觀標準（未實現損益會隨股價波動）。一般投資者最常問「我今年到底賺賠多少」這類問題，獨立 Tab + 彙總卡是最直接的回答方式。屬 P2。

**Independent Test**: 使用者於 2025 年買入 2330：第一批 2025-01-15 買 500 股 / $580、第二批 2025-03-20 買 500 股 / $620；2025-08-10 賣出 700 股 / $700。系統依 FIFO 算出：賣出 700 股對應「第一批全部 500 股 × $580 + 第二批 200 股 × $620」= 414,000 元成本基礎；實現損益 = (700 × $700 − 手續費 − 證交稅) − 414,000 = 489,000 − 698 − 1,470 − 414,000 = 72,832 元；報酬率 ≈ 17.6%。表格該筆紀錄顯示此數字。

**Acceptance Scenarios**:

1. **Given** 使用者於 2025-01-15 買 500 股 × $580、2025-03-20 買 500 股 × $620、2025-08-10 賣 700 股 × $700，**When** 觀看實現損益 Tab，**Then** 表格列出 2025-08-10 賣出該筆，成本均價 = 414,000 ÷ 700 ≈ $591、賣出均價 $700、實現損益 ≈ +72,832、報酬率 ≈ +17.6%（綠色 ▲）。
2. **Given** 使用者僅有買入未曾賣出，**When** 觀看實現損益 Tab，**Then** 彙總卡顯示總實現損益 0、報酬率 0%、已實現筆數 0；表格區呈現空狀態文案「尚無已實現損益紀錄」。
3. **Given** 同一檔股票的「未賣出剩餘部位」（如上例剩 300 股），**When** 觀看持倉頁該檔的平均成本，**Then** 平均成本以 FIFO 剩餘批次計算 = 第二批剩 300 股 × $620 ÷ 300 = $620（**非**全部買入的加權平均 $600）。
4. **Given** 使用者今年賣出 5 筆、共獲利 12,500、虧損 3,200，**When** 觀看彙總卡，**Then** 「今年實現損益」顯示 +9,300（綠色 ▲）、「已實現筆數」顯示 5。

### User Story 6 - 一鍵批次更新所有持股的最新股價（Priority: P3）

使用者點「批次更新股價」按鈕開啟 Modal，列出所有目前持股以及每檔的目前股價輸入框。可點「從證交所取得最新股價」一鍵批次拉 TWSE 最新收盤價（每檔顯示價格來源：即時／收盤／T+1 與取得時間）；也允許手動覆寫個別股價；確認後一次寫回所有股票的 `current_price` 與更新時間，整個股票頁所有市值、損益、報酬率立即重算。

**Why this priority**: 多數使用者開頁時系統已自動查價（US1 / US2 / US3 內含），但有時 TWSE API 暫時失敗或使用者想批量校準到當下盤中價格，提供一個獨立的「全量更新」工具是體驗加分項。屬 P3，因為核心查價已在其他 user story 內運作；本功能是為了「掌控感」而非「必要性」。

**Independent Test**: 使用者持有 8 檔股票，點「批次更新股價」按鈕，Modal 列出 8 列輸入框（已預填當前 `current_price`）。點「從證交所取得最新股價」後系統 8 個並發查詢，每列右側顯示「即時 $600 (13:25)」或「收盤 $602 (15:00)」或「T+1 $598 (2026-04-25)」並回填輸入框。使用者手動把第 3 檔改為 $610（覆寫），點「確認」後 8 檔的 `current_price` 全部更新，回到股票頁時市值欄位即時反映新數字。

**Acceptance Scenarios**:

1. **Given** 使用者持有 5 檔股票，**When** 開啟批次更新 Modal，**Then** Modal 列出 5 列輸入框，每列預填當前 `current_price`，並有「自動帶入最新價格」按鈕。
2. **Given** 使用者點「從證交所取得最新股價」，**When** 系統並發查詢 5 檔，**Then** 每列右側顯示價格來源（即時／收盤／T+1）與取得時間，並回填價格至輸入框；查詢失敗的個別股票顯示「查詢失敗」並保留原值。
3. **Given** 使用者手動覆寫第 2 檔為 $150（與 TWSE 帶回的 $148 不同），**When** 點「確認」，**Then** 系統以使用者輸入值為準寫入該檔 `current_price`，不被自動帶入值蓋掉。

### Edge Cases

- **零股交易**：股數 < 1000 視為零股交易；手續費套用「零股最低 1 元」而非「整股最低 20 元」；其他計算規則一致。
- **小數股數拒絕**：若使用者試圖輸入小數股數（如 1.5、0.5）系統 MUST 拒絕並顯示「股數必須為整數」；不靜默 floor 或 round 為防止帳目落差。
- **賣出超過持有股數**：若使用者試圖賣出超過目前持有股數系統 MUST 拒絕並顯示「賣出股數不可超過目前持有 (X 股)」；不允許負股數庫存。
- **TWSE API 暫時失敗**：查價失敗時 (a) 新增股票仍可成立但「目前股價」欄位為 0、`current_price` 寫入 0；(b) 個股卡片市值欄位以「—」呈現；(c) 不阻擋使用者繼續記錄交易（手動輸入價格仍可）。
- **歷史代號改名**：若 TWSE 上的股票名稱在某時點變更（如併購、分割），同步時系統 MUST 以最新查到的名稱更新 `stocks.name` 欄位，但歷史交易紀錄不變動。
- **重複股利紀錄**：同一使用者於同除息日同股票已有股利紀錄時，「同步除權息」按鈕**不**重複新增；改為跳過並於同步結果顯示「跳過 X 筆已存在」。
- **定期定額預算不足**：若「每期預算 ÷ 當前股價」< 1 股，該次排程跳過、不產生交易，並於該排程的「最近執行紀錄」標示「預算不足以買 1 股 (預算 $X / 股價 $Y)」。
- **TWSE 休市日未快取**：若 holiday API 也呼叫失敗，系統退回「僅以週末（六、日）判斷」作 fallback；雖然可能誤把彈性放假日視為交易日，但避免阻擋登入時的排程檢查。
- **多筆同日同檔股利**：若除息日當天同一檔股票同時有現金股利與股票股利（常見於台積電、鴻海等），系統 MUST 同步為一筆股利紀錄（含現金與股票兩個欄位），而非拆成兩筆。
- **註銷股票（如 6488 全營收清算下市）**：下市股票仍保留歷史交易與已實現損益紀錄；持倉股數 > 0 的下市股票於股票頁仍顯示但 `current_price` 鎖定為下市前最後一次成功取得的價格，市值以該價格計算並標示「（已下市）」。

## Requirements *(mandatory)*

### Functional Requirements

#### 持倉與總覽

- **FR-001**: 系統 MUST 為每檔持股維護「股票代號 + 類型（一般股票／ETF／權證）」資料；類型 MUST 影響賣出時的證交稅稅率：一般 0.3%、ETF / 權證 0.1%。
- **FR-002**: 持有股數、平均成本、未實現損益、報酬率等衍生數值 MUST 由系統從交易紀錄動態計算，**不**另存欄位；確保帳目永遠跟交易一致（單一資料來源）。
- **FR-003**: 投資組合總覽卡 MUST 顯示四個彙整數字：總市值、總成本、總損益（含 ▲／▼）、整體報酬率；其中總成本以 FIFO 剩餘批次計算（**非**全部買入的加權平均）。
- **FR-004**: 個股卡片 MUST 顯示八個欄位：代號、名稱、持有股數、平均成本、目前股價、市值、損益、報酬率；獲利綠色 ▲、虧損紅色 ▼；持股股數為 0 的股票**不**出現於個股卡片列表（但保留交易與已實現損益歷史）。

#### TWSE 查價整合

- **FR-005**: 系統 MUST 提供 TWSE 代理查價 API，採三段策略依「當下台灣時間」自動切換：(a) 盤中（週一至五 09:00–13:30 UTC+8）→ `mis.twse.com.tw` 即時成交價、快取 1 分鐘；(b) 盤後（週一至五 13:30 後）→ TWSE `STOCK_DAY` 當日收盤價、快取 5 分鐘；(c) 非交易日／盤前 → `STOCK_DAY_ALL`（T+1 資料）、快取 10 分鐘。
- **FR-006**: 系統 MUST 同時支援上市（TSE）與上櫃（OTC）；TWSE API 失敗時 MUST 自動嘗試對應的 TPEX 端點。
- **FR-007**: 新增股票或於交易／股利 Modal 輸入代號時 MUST 自動觸發查價並回填名稱與當前股價；查詢成功 MUST 在輸入欄旁顯示綠色提示「✓ <名稱> <價格類型> $<價>（時間/日期）」、失敗 MUST 顯示紅色提示「找不到此股票代號」；前端 MUST 套用 500ms 防抖以避免每按一鍵就打 API。
- **FR-008**: 股票代號 MUST 通過格式驗證（僅允許數字 + 少數字母，最長 8 字元）後才送出查價請求；前端與後端皆驗證以防止注入攻擊。

#### 交易與股利

- **FR-009**: 股票交易 Modal MUST 提供買入／賣出切換；切換時按鈕配色 MUST 即時變化（買入綠 / 賣出紅）；費用摘要區塊（成交金額、手續費、交易稅、總成本／淨收入）MUST 隨股數或單價變更即時重算。
- **FR-010**: 股票交易必填欄位：日期、股數（強制整數）、每股價格、扣款／入款帳戶；選填欄位：手續費（預設自動計算可手動覆寫）、交易稅（預設自動計算可手動覆寫）、備註。股數 MUST 通過 `Number.isInteger` 等同的整數檢查；小數值 MUST 拒絕並顯示「股數必須為整數」。
- **FR-011**: 手續費 MUST 預設自動計算為 `⌊成交金額 × 0.1425%⌋`（向下取整），整股交易（股數 ≥ 1000）最低 20 元；零股交易（股數 < 1000）最低 1 元；使用者 MUST 可手動覆寫此值。
- **FR-012**: 證交稅 MUST 僅於「賣出」交易計算為 `⌊成交金額 × 稅率⌋`，最低 1 元；稅率依該檔股票類型：一般 0.3%、ETF / 權證 0.1%；買入交易稅一律為 0；使用者 MUST 可手動覆寫此值。
- **FR-013**: 賣出股票交易 MUST 拒絕「賣出股數 > 目前持有股數」的提交，並顯示「賣出股數不可超過目前持有 (X 股)」；不允許負股數庫存。
- **FR-014**: 若交易 Modal 輸入的股票代號尚未建立持倉，送出交易時 MUST 自動新增該股票（從 TWSE 查名稱與股價，查詢失敗時以「（未命名）」與股價 0 建立並允許後續批次更新）。
- **FR-015**: 股利 Modal 必填欄位：除息日期（ex-dividend date）、股票代號、入款帳戶；至少需填現金股利每股或股票股利每千股配股數其中一項。日期欄位 MUST 統一稱為「除息日期」（不混用「入帳日期」），因其為決定資格的法定基準日。
- **FR-016**: 股利金額計算：現金股利金額 = 持股數 × 每股現金股利（四捨五入為整數）；股票股利股數 = 持股數 × 每千股配股數 ÷ 1000（無條件捨去為整數）；股票股利 MUST 增加該檔的持有股數（透過寫入一筆「股票股利」型交易紀錄）。

#### 列表與分頁

- **FR-017**: 交易紀錄頁與股利紀錄頁 MUST 支援：(a) 依股票代號或名稱的下拉式搜尋篩選；(b) checkbox 多選 + 批次刪除；(c) 每頁筆數可選（預設 20，提供 10 / 20 / 50 / 100 與自訂）；(d) 伺服器端分頁（response 包含 `data, total, page, totalPages`）。
- **FR-018**: 批次刪除 MUST 經過二次確認對話框，列出將刪除的筆數；刪除後對應持股的衍生數值（持有股數、平均成本、損益、實現損益）MUST 立即重算。

#### 定期定額

- **FR-019**: 系統 MUST 允許使用者為某檔股票設定定期定額排程，欄位：股票代號、每期預算（TWD）、週期（每週／每月／每季）、起始日、扣款證券帳戶、備註、啟用／停用 flag；同一檔股票 MUST 允許多筆排程並存（不設 (user_id, stock_id) 唯一鍵）。
- **FR-020**: 定期定額觸發點 MUST 為「使用者每次登入時」進行檢查（沿用既有 004 / 005 的 server-side 觸發 pattern，**不**引入獨立 cron 服務）；檢查時對每筆 `enabled = 1` 排程比對「上次執行 < 本期起點」即觸發。
- **FR-021**: 定期定額觸發時系統 MUST 依「每期預算 ÷ 當前股價」計算可買股數（向下取整）；若可買股數 < 1 則跳過該次並於該排程的「最近執行紀錄」標示「預算不足以買 1 股 (預算 $X / 股價 $Y)」，**不**產生交易。
- **FR-022**: 若排程日恰逢週末或 TWSE 休市日，系統 MUST 自動順延至下一個交易日；交易紀錄日期寫入實際交易日，備註附「原排程 YYYY-MM-DD 順延」；排程節奏不變（下次仍以原起始日推算）。
- **FR-023**: TWSE 休市日資料 MUST 從 TWSE `/v1/holidaySchedule/holidaySchedule` OpenAPI 取得，過濾掉「開始交易／最後交易」等特別交易日後快取 24 小時；若該 API 也失敗，系統 MUST 退回「僅以週末（週六、週日）判斷」作 fallback，避免阻擋登入時的排程檢查。
- **FR-024**: 排程被使用者停用後再重新啟用時，系統 MUST **不**追溯補產生停用期間漏掉的交易；MUST 僅從下一次自然觸發點開始（與 005 排程信件「停用→啟用不補寄」原則一致）。

#### 自動同步除權息

- **FR-025**: 股利紀錄頁 MUST 提供「同步除權息」按鈕；點下後後端依該使用者最早交易日期到今日，**按年分段**查 TWSE `TWT49U` 除權息列表（避免單次撈取資料量過大）。
- **FR-026**: 同步邏輯 MUST 僅處理該使用者已持有過的股票代號（包含目前持股 + 歷史持股），且依除息日當下的持股數計算金額（持股 = 0 跳過）；含股票股利者 MUST 額外查 `TWT49UDetail` 明細取每股現金與每千股配股數。
- **FR-027**: 同步結果 MUST 依下列規則寫入：(a) 同除息日同股票已存在股利紀錄則跳過、不重複新增；(b) 自動新增的紀錄備註 MUST 標示「TWSE自動同步（每股$X.XX）」；(c) 同步完成 MUST 顯示彙總（新增 N 筆 / 跳過 M 筆 / 失敗 K 筆）。
- **FR-028**: 除權息 API 結果 MUST 快取 30 分鐘以避免短時間內重複呼叫。

#### 實現損益（FIFO）

- **FR-029**: 系統 MUST 提供「實現損益」獨立 Tab；頂部彙總卡 MUST 顯示四個數字：總實現損益（含 ▲／▼）、整體報酬率、今年實現損益、已實現筆數。
- **FR-030**: 實現損益 MUST 採 FIFO（先進先出）邏輯計算：(a) 買入批次依時間序進入佇列，每批次的成本基礎包含手續費分攤；(b) 每筆賣出依序從最早批次扣除；(c) 實現損益 = 賣出收入（股數 × 賣出價 − 手續費 − 證交稅）− FIFO 成本基礎；(d) 報酬率 = 實現損益 ÷ FIFO 成本基礎 × 100%。
- **FR-031**: 未賣出剩餘部位的「平均成本」MUST 以 FIFO 剩餘批次計算（**非**全部買入的加權平均）；該值為 US1 個股卡片「平均成本」欄位的單一資料來源。
- **FR-032**: 實現損益表格 MUST 列出每筆賣出交易的：賣出日期、股票（代號 + 名稱）、股數、賣出均價、成本均價（FIFO）、手續費 + 稅、實現損益、報酬率；可點欄位排序（預設依賣出日期由新到舊）。

#### 批次更新股價

- **FR-033**: 系統 MUST 提供「批次更新股價」Modal，列出所有目前持股（持股股數 > 0）以及每檔的目前股價輸入框（預填當前 `current_price`）。
- **FR-034**: Modal MUST 提供「從證交所取得最新股價」按鈕，按下後系統並發查詢所有持股的最新股價（採 FR-005 三段策略）；每列右側 MUST 顯示價格來源（即時／收盤／T+1）與取得時間；查詢失敗的個別股票 MUST 顯示「查詢失敗」並保留原值。
- **FR-035**: 使用者 MUST 可手動覆寫個別股價（例如想用即時盤中價而非 T+1 收盤）；點「確認」後系統 MUST 以使用者輸入值為準寫回所有持股的 `current_price` 與更新時間，整個股票頁所有市值、損益、報酬率 MUST 立即重算。

#### 跨頁共用

- **FR-036**: 所有金額（持倉、交易、股利、實現損益）MUST 以 TWD 為單位顯示與計算；本功能僅支援台股（TSE / OTC），不提供海外股票（與 005 的「不提供多幣別報表切換」原則一致）。
- **FR-037**: 持倉、交易、股利、實現損益的所有衍生數值 MUST 在資料變動後（新增 / 修改 / 刪除任一筆交易或股利）重新進入頁面時即時反映變動，不依賴額外手動重整或快照凍結（與 005 SC-006 即時性原則一致）。

### Key Entities *(include if feature involves data)*

- **股票（Stock）**：使用者持有的台股單一代號，含代號、名稱、類型（一般／ETF／權證）、目前股價、最後查價時間。持有股數、平均成本、損益等衍生欄位**不**儲存，由交易紀錄動態計算。
- **股票交易（Stock Transaction）**：一筆買入或賣出紀錄，含交易日期、股票、買賣方向、股數（整數）、每股價格、手續費、證交稅、扣款／入款帳戶、備註。買入會增加 FIFO 佇列尾端、賣出會從佇列頭部依序扣除。
- **股利（Stock Dividend）**：一筆現金股利或股票股利紀錄，含除息日期（ex-dividend date）、股票、現金股利金額、股票股利股數、入款帳戶、備註、來源（手動／TWSE 自動同步）。股票股利同時寫入一筆「股票股利」型交易以維持 FIFO 佇列正確性。
- **定期定額排程（Stock Recurring Plan）**：使用者為某檔股票設定的長期投資排程，含股票、每期預算、週期、起始日、扣款帳戶、備註、啟用狀態、最後執行時間、最後執行摘要。
- **TWSE 休市日（TWSE Trading Holiday）**：從 TWSE 公開 API 同步的非交易日清單，過濾「開始交易／最後交易」等特別交易日後快取 24 小時；用於定期定額排程的順延邏輯。
- **FIFO 成本批次（FIFO Cost Lot）**：buildtime 計算中介資料，非持久化；某筆買入交易進入佇列後即視為一個批次（含原始股數、剩餘股數、成本均價含手續費分攤）；賣出依時間序從佇列頭部扣除以決定該筆賣出的成本基礎。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者開啟股票頁後，投資組合總覽卡與所有個股卡片 MUST 在 2 秒內完整呈現於可視範圍內（持股數 ≤ 50 檔的常見場景）。
- **SC-002**: 使用者於交易 Modal 輸入股數或單價時，費用摘要 MUST 在 200ms 內即時重算並更新；輸入股票代號後 TWSE 查價結果 MUST 在 1.5 秒內回傳並回填欄位（90% 的查詢達到此標準，受 TWSE 服務延遲影響）。
- **SC-003**: 同步除權息按鈕觸發後，含 5 年歷史交易與 20 檔不同股票的使用者 MUST 在 30 秒內完成全量同步（含 TWSE API 呼叫 + 持股計算 + 寫入）；同步結果 MUST 顯示彙總（新增 / 跳過 / 失敗筆數）。
- **SC-004**: FIFO 實現損益計算 MUST 100% 一致於券商實際稅務認定方式（給定相同交易序列，本系統算出的成本基礎與台灣本土主要券商 App 之顯示誤差 ≤ 1 元，差異僅源於四捨五入策略）。
- **SC-005**: 定期定額排程觸發 MUST 在使用者登入後 3 秒內完成檢查與必要的買入交易產生（持股 < 50 檔、排程 < 20 筆的常見場景）；不阻擋使用者操作 UI（採非同步背景執行）。
- **SC-006**: 使用者新增、修改、刪除一筆交易或股利後重新進入股票頁，所有受影響的持股、平均成本、損益、實現損益 MUST 反映該筆變動，無快照延遲（與 004 SC-006 / 005 SC-006 即時性原則一致）。
- **SC-007**: 90% 的使用者於股票頁停留少於 30 秒即可回答「我目前帳上賺多少／賠多少」這類自我提問（透過總覽卡 + 個股卡片組合即足以判斷）。*（此項屬 post-launch user retention metric，需透過分析工具或使用者訪談量測，本功能 build-time 不直接驗收；列入此處以對齊產品目標。）*

## Assumptions

- **沿用 004 / 005 的「登入時觸發」pattern**：定期定額排程於每次登入時 server-side 檢查；本功能不引入獨立 cron 服務（與 005 排程信件不同 — 005 沿用既有 setInterval 5 分鐘 cron tick）。
- **沿用既有 TWSE 三段查價策略**：FR-005 ~ FR-008 直接複用既有股票模組已實作的查價邏輯（盤中 mis / 盤後 STOCK_DAY / 非交易日 STOCK_DAY_ALL），本功能規範其行為但不重新設計。
- **沿用既有 transaction 資料模型**：股票交易與股利紀錄沿用既有 `stock_transactions` / `stock_dividends` 表，不引入新表（除非 005 之外的 schema 補強需要）。
- **TWSE 休市日 OpenAPI 穩定可用**：FR-023 假設 TWSE `/v1/holidaySchedule/holidaySchedule` OpenAPI 持續可用；若 TWSE 改版下線則需於該事件中重新規劃。
- **FIFO 為法定預設**：台灣個人投資者證所稅依 FIFO 認定；本功能不提供 LIFO / 加權平均選項，與政府預設一致。
- **手續費率採台灣標準 0.1425% / 證交稅率 0.3%（一般）/ 0.1%（ETF / 權證）**：依台灣現行法規；若未來費率變更則需於該事件中更新硬編值（短期內不引入「動態費率設定」UI）。
- **排程器以「自然週／月／季」為單位**：每週 = 每週一觸發；每月 = 每月 1 號觸發；每季 = 每季首月 1 號觸發；起始日後第一次觸發於對齊週期的下一個週期起點。
- **不引入新 npm 套件、不引入新前端 CDN、不引入新外部 API**：與 004 / 005 一致的「零新依賴」治理原則；本功能於既有 stack（Node.js 24 + Express 5 + sql.js + Chart.js + decimal.js）上實作。

## Out of Scope

- 海外股票（美股、港股、日股等）— 本功能僅限台股（TSE 上市 + OTC 上櫃）
- 現股當沖證交稅減半 — 預設皆以完整稅率計算（一般 0.3% / ETF 0.1%），留給未來版本
- 股利再投資（DRIP）自動追蹤 — 使用者收到股利後若想再投資需手動建立買入交易
- 歷史股價走勢圖表 — 本功能僅顯示當前股價與市值；K 線、分時走勢留給未來版本
- 期貨、選擇權、債券、ETN 等其他金融商品 — 僅限台股股票（含 ETF）
- 多幣別股票（如複委託美股以美元計價）— 與 005 「TWD 統一計價」原則一致
- LIFO / 加權平均成本法 — FIFO 為法定預設，不提供切換
