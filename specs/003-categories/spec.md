# Feature Specification: 分類系統（Category System）

**Feature Branch**: `003-categories`
**Created**: 2026-04-25
**Status**: Draft
**Input**: User description: "分類系統 — 兩層階層（父分類／子分類）、預設分類樹、收支類型約束、顏色／排序屬性，畫面以父分類整列＋子分類縮排網格呈現，含完整刪除與循環約束"

## Clarifications

### Session 2026-04-25

- Q: 父分類本身能否被指派為某筆交易的分類？ → A: 否 — 交易僅能指派給子分類（leaf-only）；父分類不出現在新增／編輯交易的分類選單，僅作為彙整維度。
- Q: 自動補建預設子分類時，若使用者已自行建立同父下同名的子分類，怎麼處理？ → A: 跳過補建 — 以「同父下同名」視為該預設項已存在，不重建、不修改使用者既有項的旗標或屬性。
- Q: 「是否為預設」旗標除了補建判定外是否帶有任何行為差異？ → A: 否 — 純資訊／追溯用途，使用者對預設分類的編輯、改名、改色、刪除規則皆與一般分類完全相同。
- Q: 子分類能否在父分類之間移動，移動後既有交易如何處理？ → A: 可移動 — 編輯子分類時可改其所屬父分類，既有交易仍掛在該子分類，彙整自動跟隨新父；移動時驗證收支類型一致、不形成循環、同新父下不重名。
- Q: 父分類的唯一性規則為何？ → A: 唯一性鍵為 `(user_id, type, name)` — 同收支類型內父分類名稱不可重複；跨類型可同名（如支出/其他、收入/其他並存）。

### Session 2026-04-25 (round 2)

- Q: 收入分類與「其他」父分類在 leaf-only 規則下無預設子分類即不可用，如何處理？ → A: 為每個父分類重新設計一組合理的預設子分類（含原本已有的也一併重做），確保所有預設父分類在註冊後立即可用、且全部維持嚴格兩層樹。詳見 FR-008 修訂版。
- Q: 升級補建的時機與登入流程關係？ → A: 同步補建 — 在登入回應前完成；P95 延遲 ≤ 200 ms；補建失敗仍允許登入並於下次登入重試（冪等保證）。
- Q: 排序順序的編輯互動方式？ → A: 拖曳排序（drag-and-drop）— 使用者直接拖曳父分類列或子分類網格項目調整順序；後端提供「重排同層分類」的批次 API。
- Q: 補建是否會「復活」使用者主動刪除的預設分類？ → A: 否 — 系統維護每使用者的「已刪除預設項」清單，補建時跳過此清單；分類管理頁提供「還原預設分類」操作可清空清單，使下次登入時重新補建。
- Q: 「還原預設分類」操作的範圍與副作用？ → A: 僅補回缺漏 — 清空 DeletedDefaultRegistry 並執行 FR-010 補建；既有分類（含自訂與已客製化的預設項）完全不動；UI 文案應明確為「補回過去刪除的預設分類」而非「重置」。

### Session 2026-04-25 (round 3)

- Q: 編輯分類能否變更收支類型（income↔expense）？ → A: 全面禁止 — 任何分類（父或子）建立後其 type 永久不可變；UI 將 type 欄位顯示為 read-only。誤建錯類型必須走「移光交易 → 刪除 → 重建」流程修正。
- Q: 連帶刪除的預設子分類是否進入 DeletedDefaultRegistry？ → A: 對稱寫入 — 刪除父分類連帶刪除子分類時，每個被連帶刪除的預設子分類都加入 registry，與「逐一刪除」效果完全等效。
- Q: 分類管理頁的收入/支出組織方式？ → A: 雙區塊 — 同一頁從上到下兩個區塊，先「支出」（含其全部父分類列+子分類網格），再「收入」；中間以區塊標題與分隔線區分，不用 tab 切換。
- Q: 子分類移到另一父分類後的 sort_order 規則？ → A: 移到末端 — `sort_order` 自動設為新父底下既有子分類最大值 + 1，放到該父底下最後位置；使用者若要再排序，用 FR-024a 的拖曳。
- Q: 父分類「隱藏」狀態的語意（是否連帶影響子分類）？ → A: 移除「是否隱藏」整個功能 — 父分類與子分類皆不設計 hidden 屬性；不再支援「隱藏分類」此種狀態。使用者若不再需要某分類，僅能透過刪除（FR-017～FR-019）處理。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 註冊即擁有完整可用的分類樹（Priority: P1）

新使用者完成註冊後，無需手動建立任何分類即可立即開始記帳：系統已自動為其建立一棵涵蓋日常生活的預設分類樹（餐飲、交通、購物、娛樂、居住、醫療、教育、其他、以及收入類），每個父分類下都有預先準備好的子分類與專屬顏色，分類顏色與儀表板、報表的圖例完全一致。

**Why this priority**: 這是分類系統能否「對使用者有用」的根基。沒有預設樹，使用者第一次打開記帳頁就要先做設定工作；有了預設樹，使用者第一筆交易就能直接落入有意義的分類，後續的儀表板與報表也立刻有顏色一致的視覺語言可用。本系統其他所有功能（新增交易、報表、編輯分類）都建立在「分類已存在」的前提上。

**Independent Test**: 註冊一個全新帳號並登入，不做任何手動分類設定，直接進入分類管理頁，可看到完整的預設父／子分類樹；進入新增交易頁，分類選擇器中預設分類可立即選用；新增任一筆交易後，儀表板的圖例顏色與分類管理頁的顏色完全一致。

**Acceptance Scenarios**:

1. **Given** 一個新註冊的使用者第一次登入，**When** 進入分類管理頁，**Then** 可看到 8 個支出父分類（餐飲、交通、購物、娛樂、居住、醫療、教育、其他）與 5 個收入分類（薪資、獎金、投資、兼職、其他），且餐飲、交通、購物、娛樂、居住、醫療、教育下方有對應的預設子分類。
2. **Given** 預設分類已建立，**When** 使用者新增一筆「午餐」交易並回到儀表板，**Then** 該交易的子分類顏色與分類管理頁顯示的顏色一致，且歸屬於「餐飲」父分類的彙整。
3. **Given** 一位舊使用者曾在系統升級前手動刪除過某個預設子分類，**When** 該使用者下次登入，**Then** 系統偵測到缺少的預設子分類並自動補建，但不覆寫使用者已自訂的分類；自訂分類保持不變。

### User Story 2 - 用兩層階層組織自己的分類（Priority: P1）

使用者可以在預設分類樹之上，依自己直覺新增父分類與子分類、調整顏色、排序，並重新命名既有分類；操作介面讓父分類整列展開、子分類以縮排網格佈局呈現，便於一眼掌握整棵樹。

**Why this priority**: 預設樹只是起點；分類系統的價值在於每個使用者都能依自己的記帳語彙微調樹形。沒有編輯能力，使用者很快會抱怨「分類不符合我的習慣」並放棄系統。階層被刻意限制為兩層也是核心設計決策——必須在 P1 就被驗證可用。

**Independent Test**: 在已有預設分類的帳號中，新增一個名為「寵物」的父分類（支出）、給予顏色與排序、再新增「飼料」「醫療」「美容」三個子分類；將「3C 產品」重新命名為「電子產品」；所有變更立即在分類管理頁與新增交易頁的下拉選單中生效。

**Acceptance Scenarios**:

1. **Given** 使用者在分類管理頁，**When** 點擊「新增父分類」並輸入名稱、選擇收支類型與 `#RRGGBB` 顏色，**Then** 父分類以整列寬度顯示在指定排序位置，並帶有 `+`（新增子分類）與編輯按鈕。
2. **Given** 使用者在某個父分類旁點擊 `+`，**When** 輸入子分類名稱，**Then** 子分類以縮排網格佈局顯示在該父分類下方，帶左側藍色邊框與箭頭圖示，且其收支類型自動繼承自父分類。
3. **Given** 使用者嘗試在「餐飲」父分類下建立第二個名為「午餐」的子分類，**When** 提交，**Then** 系統拒絕並顯示「同一父分類下名稱不可重複」的錯誤訊息，原本的「午餐」保持不變。
4. **Given** 一個父分類為「支出」類型，**When** 使用者嘗試在其下建立「收入」類型的子分類，**Then** 系統拒絕並提示子分類類型必須與父分類一致。
5. **Given** 使用者在編輯一個父分類的對話框，**When** 嘗試將「父分類」欄位改為某個現有分類（亦即試圖把父分類改成另一分類的子分類），**Then** 系統拒絕該變更以防止形成循環階層。

### User Story 3 - 安全地刪除不再需要的分類（Priority: P2）

使用者可以刪除自己不需要的子分類或父分類，但系統會保護尚有交易紀錄的分類不被誤刪，使歷史報表保持完整。

**Why this priority**: 刪除是低頻但必要的能力，且具有資料完整性風險。在 P1 已驗證新增與編輯後，P2 補上安全刪除即可形成完整的 CRUD 閉環。系統不提供「隱藏」功能（見 Clarifications round 3），使用者唯一停用某分類的方式即為刪除，因此刪除規則的清晰度直接影響使用者信任。

**Independent Test**: 嘗試刪除一個尚有交易的子分類，系統拒絕並顯示理由；先把該分類下所有交易改掛到其他分類後再刪除成功；刪除一個整棵子樹下都沒有交易的父分類，子分類連帶被刪除；嘗試刪除一個父分類，但其底下其中一個子分類有交易紀錄，整棵樹不可刪除。

**Acceptance Scenarios**:

1. **Given** 一個子分類底下有 1 筆以上的交易，**When** 使用者嘗試刪除該子分類，**Then** 系統拒絕並提示「該分類下尚有交易紀錄，無法刪除」，分類維持原狀。
2. **Given** 一個父分類，其下任一子分類仍有交易紀錄，**When** 使用者嘗試刪除該父分類，**Then** 系統拒絕整棵樹的刪除並提示哪些子分類仍有交易。
3. **Given** 一個父分類，其下所有子分類都沒有任何交易紀錄，**When** 使用者確認刪除該父分類，**Then** 父分類與其所有子分類一併被刪除。
4. **Given** 使用者刪除任一分類，**When** 操作完成，**Then** 分類管理頁、交易表單分類選單與儀表板圖例同步更新，不再顯示已刪除的分類。

### Edge Cases

- 顏色欄位若被傳入非 `#RRGGBB` 格式（例如 `red`、`#ABC`、`<script>`），後端必須拒絕並回傳驗證錯誤；前端表單應在送出前阻擋並提示。
- 兩個使用者各自建立同名分類「自訂」彼此互不影響；分類的唯一性僅限於同一使用者、同一父分類底下。
- 預設分類雖標記為「預設」，仍允許使用者重新命名、改色；刪除預設分類遵循與一般分類相同的「無交易才能刪」規則。
- 排序順序欄位需支援使用者拖曳重新排序父分類與某父分類底下的子分類；若兩筆紀錄排序相同，以建立時間或 ID 作為穩定次序。
- 系統升級後執行的補建動作必須具冪等性（同一使用者多次登入只補建一次），且不得覆蓋使用者對既有預設分類所做的客製化（顏色、名稱）。
- 已被刪除的分類在歷史報表與既有交易明細中如何顯示，由報表規格決定（本規格不規範）；本規格僅保證刪除規則（FR-017～FR-019）能保護所有有交易的分類不被誤刪。

## Requirements *(mandatory)*

### Functional Requirements

#### 階層與資料模型

- **FR-001**: 系統 MUST 支援恰好兩層分類階層：父分類（root）與子分類（leaf）。系統 MUST 拒絕建立第三層（孫分類）。
- **FR-002**: 系統 MUST 為每個分類儲存以下屬性：名稱、收支類型（收入或支出）、顏色（`#RRGGBB`）、是否為預設分類、排序順序、所屬使用者；子分類 MUST 額外記錄其父分類的關聯。系統 MUST NOT 設計或儲存「是否隱藏」屬性。
- **FR-003**: 系統 MUST 強制每個分類的收支類型只能為「收入」或「支出」其中之一（資料層 CHECK 約束）。
- **FR-004**: 系統 MUST 強制子分類的收支類型與其父分類一致；不允許建立或編輯成不一致的狀態。
- **FR-005**: 系統 MUST 強制同一父分類底下的子分類名稱不可重複（同使用者範圍內）。
- **FR-005a**: 系統 MUST 強制父分類在 `(user_id, type, name)` 範圍內唯一：同一使用者、同一收支類型下，父分類名稱不可重複；跨收支類型則允許同名（例如支出類「其他」與收入類「其他」可並存）。
- **FR-006**: 系統 MUST 強制每個子分類僅能隸屬於唯一一個父分類。

#### 預設分類樹

- **FR-007**: 系統 MUST 在新使用者註冊時自動建立完整的預設分類樹，包含以下支出父分類：餐飲、交通、購物、娛樂、居住、醫療、教育、其他；以及以下收入分類：薪資、獎金、投資、兼職、其他。
- **FR-008**: 系統 MUST 為「每一個」預設父分類（含支出與收入）建立至少一個預設子分類，使預設樹在 leaf-only 規則（FR-013a）下對所有父分類皆立即可用。完整預設子分類集如下：

  支出（Expense）：
  - 餐飲 → 早餐、午餐、晚餐、飲料、點心
  - 交通 → 大眾運輸、計程車、加油、停車費、高鐵/火車
  - 購物 → 日用品、服飾、3C用品、家電、美妝保養
  - 娛樂 → 電影/影音、遊戲、旅遊、運動健身、訂閱服務
  - 居住 → 房租/房貸、水電費、瓦斯費、網路費、管理費
  - 醫療 → 掛號費、藥品、保健食品、牙科、健檢
  - 教育 → 學費、書籍、線上課程、補習費
  - 其他 → 雜支、禮金/紅包、捐款、罰款

  收入（Income）：
  - 薪資 → 月薪、加班費
  - 獎金 → 年終獎金、績效獎金、節日禮金
  - 投資 → 股利、利息、資本利得
  - 兼職 → 接案、家教、打工
  - 其他 → 退稅、贈與/紅包、雜項
- **FR-009**: 系統 MUST 在每個預設分類上標記「預設分類」旗標，並指派一致的預設顏色，使儀表板與報表的圖例配色從第一筆交易起即穩定。
- **FR-009a**: 「是否為預設」旗標 MUST 為純資訊性質：除 FR-010 補建判定外，系統 MUST NOT 因該旗標而對使用者的編輯、改名、改色或刪除施加任何額外限制；預設分類所有操作行為與一般分類完全一致。
- **FR-010**: 系統 MUST 在使用者每次登入時，檢查當前定義的預設分類樹中是否有缺漏項目；若有，MUST 自動補建缺漏的預設分類，且補建必須具冪等性。
- **FR-010a**: 補建 MUST 在登入回應前同步完成，使使用者登入後立即看到完整的預設樹；補建步驟（含查詢 + 必要 INSERT）的 P95 延遲 MUST ≤ 200 ms。
- **FR-010b**: 若補建因任何原因失敗，系統 MUST 仍讓登入流程成功完成（避免阻擋使用者），並 MUST 記錄錯誤；下次登入時依 FR-010 冪等重試。
- **FR-011**: 自動補建 MUST NOT 覆寫使用者既有的客製化（含名稱、顏色、排序）；僅補建完全不存在的預設分類。
- **FR-011a**: 補建演算法 MUST 以 `(user_id, parent_category, name)` 為存在性判定鍵：當某預設子分類的目標名稱在對應父分類下已被使用者佔用，系統 MUST 跳過該項補建，且 MUST NOT 修改既有項目的任何屬性（包含 `is_default` 旗標）。
- **FR-011b**: 系統 MUST 為每位使用者維護一份「已刪除預設項」清單（以預設分類的穩定識別鍵記錄，例如 `(type, parent_default_name, default_name)` 對應子分類、`(type, default_name)` 對應父分類）。當使用者刪除一個預設分類時，MUST 將其加入該清單。
- **FR-011b1**: 當使用者透過 FR-019 刪除父分類並連帶刪除其底下所有子分類時，系統 MUST 將該父分類及**每一個被連帶刪除的預設子分類**逐一加入 `DeletedDefaultRegistry`，使「逐一刪除整棵樹」與「直接刪除父分類連帶整樹」兩種操作對 registry 的最終狀態完全一致。
- **FR-011c**: 補建演算法（FR-010）執行時 MUST 跳過已存在於該使用者「已刪除預設項」清單中的所有項目；使用者主動刪除過的預設分類 MUST NOT 被自動重建。
- **FR-011d**: 分類管理頁 MUST 提供「還原預設分類」操作；觸發後系統 MUST 清空該使用者的「已刪除預設項」清單，並 MUST 立即執行一次 FR-010 的補建程序，將清單清空後缺漏的所有預設分類補回。
- **FR-011e**: 「還原預設分類」操作 MUST 為「非破壞性」：MUST NOT 刪除、覆寫或修改任何已存在的分類（無論其為自訂分類或已被使用者客製化的預設分類），也 MUST NOT 變更任何分類的顏色、名稱、sort_order；該操作的可觀察結果僅限於「`DeletedDefaultRegistry` 被清空」與「過去缺漏的預設項被新增回來」兩件事。
- **FR-011f**: 「還原預設分類」按鈕的 UI 文案 MUST 明確表達其「補回過去刪除的預設分類」之語意，禁止使用「重置」「恢復出廠」等可能誤導使用者以為會清除客製化的字眼。

#### CRUD 操作

- **FR-012**: 使用者 MUST 能新增父分類，提供名稱、收支類型、顏色，並可指定排序順序。
- **FR-013**: 使用者 MUST 能在任一父分類下新增子分類；子分類的收支類型 MUST 直接繼承自父分類，使用者不需也不能在新增時指定。
- **FR-013a**: 系統 MUST 僅允許交易指派給子分類（leaf-only）；父分類 MUST NOT 出現在新增／編輯交易的分類選單中，僅作為儀表板與報表的彙整維度。
- **FR-014**: 使用者 MUST 能編輯任一分類的名稱、顏色、排序順序。
- **FR-014c**: 分類的收支類型（income/expense）一經建立 MUST 為不可變（immutable）：系統 MUST 在編輯介面將 type 欄位顯示為 read-only，並在後端 API 對任何試圖變更 type 的請求一律拒絕。誤建錯誤類型的修正 MUST 透過「移光該分類底下所有交易 → 刪除分類 → 在正確 type 下重建」三步流程完成。
- **FR-014a**: 使用者 MUST 能將既有子分類改掛到另一個收支類型相同的父分類；變更時系統 MUST 重新驗證 FR-004（類型一致）、FR-005（同新父下名稱不重複）、FR-015（不形成循環）。
- **FR-014b**: 子分類在父分類之間移動時，原本掛在該子分類底下的交易 MUST 維持原本的 category 參照不變；其在儀表板與報表上的「父分類彙整」自動跟隨新父分類，無需批次更新交易。
- **FR-014d**: 子分類移到另一父分類時，系統 MUST 自動將該子分類的 `sort_order` 設為「新父分類底下所有既有子分類 `sort_order` 的最大值 + 1」，使其落在新父底下最後一個位置；使用者後續可透過 FR-024a 拖曳調整。
- **FR-015**: 編輯父分類時，系統 MUST 拒絕將其變更為任何分類的子分類（避免循環階層）。
- **FR-016**: 編輯時系統 MUST 重新驗證 FR-003、FR-004、FR-005、FR-006 的所有約束。

#### 刪除規則

- **FR-017**: 系統 MUST 拒絕刪除任何「底下尚有 1 筆以上交易紀錄」的分類，並回傳清楚的錯誤訊息。
- **FR-018**: 刪除父分類時，系統 MUST 檢查該父分類底下所有子分類；若任一子分類仍有交易紀錄，MUST 拒絕整棵樹的刪除。
- **FR-019**: 刪除父分類且其底下所有子分類皆無交易時，系統 MUST 連帶刪除該父分類與其所有子分類（一次原子性操作）。

#### 顏色與安全

- **FR-020**: 系統 MUST 僅允許 `#RRGGBB` 格式的六位十六進位顏色（含開頭 `#`，共 7 字元）；任何其他格式（含縮寫 `#RGB`、命名色、含 HTML/CSS 注入的字串）後端 MUST 拒絕。
- **FR-021**: 顏色驗證 MUST 在後端執行（不僅前端）；目的明確為防止 CSS 注入攻擊。

#### 顯示與排序

- **FR-022**: 分類管理頁 MUST 將父分類以整列寬度顯示，每列帶有「新增子分類」（`+`）與「編輯」按鈕。
- **FR-022a**: 分類管理頁 MUST 將分類依收支類型分為兩個垂直區塊，**先「支出」區塊、後「收入」區塊**；每個區塊內含其所有父分類列與子分類網格；兩區塊以區塊標題（明示「支出」「收入」）與分隔線區分；不使用 tab 切換。同一區塊內的父分類順序仍依 FR-024 排序欄位。
- **FR-023**: 分類管理頁 MUST 將子分類以縮排的網格佈局顯示在所屬父分類下方，帶左側藍色邊框與箭頭圖示，使視覺上能立即辨識階層關係。
- **FR-024**: 分類顯示順序 MUST 依排序順序欄位由小到大排列；同排序值時以穩定次序（如建立時間或 ID）排列。
- **FR-024a**: 使用者 MUST 能透過拖曳（drag-and-drop）調整同層分類的排序：拖曳父分類列可重排父分類、拖曳子分類網格項目可重排該父分類底下的子分類。系統 MUST 提供「批次重排同層分類」的後端操作，使一次拖曳結束後同層所有受影響項目的 `sort_order` 在同一原子性操作中更新。
- **FR-024b**: 拖曳 MUST 僅作用於同層、同父分類範圍：拖曳子分類**不能跨父**改變歸屬（跨父歸屬變更走 FR-014a 的「編輯子分類 → 改父分類」路徑）。
- **FR-025**: 分類使用的顏色 MUST 在分類管理頁、新增／編輯交易表單、儀表板、所有報表中保持完全一致。
- **FR-026**: ~~（保留編號，已移除）~~ 系統不設計「是否隱藏」屬性；新增／編輯交易的分類選單 MUST 列出該使用者所有未刪除的子分類（依 FR-013a leaf-only），不存在「隱藏排除」概念。

#### 範圍外（Non-Goals）

- **FR-027**: 系統 MUST NOT 提供 AI 自動分類或分類建議功能。
- **FR-028**: 系統 MUST NOT 提供跨使用者共用的分類模板庫；使用者間共享分類設定僅能透過 CSV 匯入／匯出（CSV 功能本身不在本規格範圍內）。

### Key Entities *(include if feature involves data)*

- **Category（分類）**：使用者用來組織收支的分類節點。屬性包含：名稱、收支類型（income / expense）、顏色（`#RRGGBB`）、是否為預設、排序順序、所屬使用者。一個分類要嘛是父分類（無 parent 關聯），要嘛是子分類（恰好指向一個父分類）。本實體不含「是否隱藏」屬性。
- **CategoryTree（分類樹，邏輯概念）**：屬於同一使用者、同一收支類型的所有分類所構成的兩層樹；每個使用者擁有獨立的分類樹，互不可見、互不影響。
- **DefaultCategoryDefinition（預設分類定義，系統內部）**：系統層級的常數，定義新使用者註冊時與舊使用者登入補建時要建立的預設父／子分類及其顏色、預設排序順序、以及供 FR-011b 使用的穩定識別鍵。此定義不暴露為使用者可編輯的資料。
- **DeletedDefaultRegistry（已刪除預設項清單，每使用者）**：每位使用者一份，記錄其曾主動刪除過的預設分類識別鍵集合。供 FR-011c 在補建時跳過、供 FR-011d 在「還原預設分類」操作時清空。此資料屬於使用者自身，不跨使用者共用。
- **與 Transaction（交易）的關聯**：每筆交易參照恰好一個子分類（leaf Category，依 FR-013a）；父分類本身永遠不會被交易直接參照。「父分類底下是否尚有交易」的判定，等同於「該父分類所擁有的所有子分類加總後是否仍有交易」（FR-018）；「子分類底下是否尚有交易」則直接查該子分類的交易（FR-017）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% 的新使用者在註冊完成的下一秒，分類管理頁即顯示完整的預設父／子分類樹（8 個支出父分類、5 個收入分類、以及 FR-008 條列的所有預設子分類），且**每一個預設父分類底下都至少有 1 個預設子分類**，使新使用者在 leaf-only 規則下立即能新增任何類型的交易。
- **SC-002**: 使用者完成「新增一個父分類並在其下新增 3 個子分類」的完整操作 MUST 能在 90 秒內完成（介面流暢度與步驟數的衡量基準）。
- **SC-003**: 100% 嘗試違反核心約束的操作（建立第三層、子分類類型不一致、同父下重名、刪除有交易的分類、循環階層、非 `#RRGGBB` 顏色）皆會被系統拒絕，且使用者收到可理解的錯誤訊息。
- **SC-004**: 任意一筆交易在儀表板、報表、分類管理頁所顯示的分類顏色完全相同（視覺一致性 100%）。
- **SC-005**: 系統升級後，所有既有使用者於下次登入時都被冪等地補齊缺漏的預設子分類；同一使用者連續登入 N 次後，預設分類補建動作只會在第一次實際發生變更，後續登入皆為無操作（idempotent）；且使用者主動刪除過的預設項（記錄於 `DeletedDefaultRegistry`）絕不被自動重建（FR-011c）。
- **SC-007**: 補建程序（FR-010）的 P95 延遲 ≤ 200 ms（同步執行於登入回應前），且即使補建失敗登入仍 100% 成功完成（FR-010b）。
- **SC-006**: 99% 以上的使用者於第一次使用分類管理頁時，能在不參考說明文件的情況下正確區分父分類列與子分類網格的階層關係（透過版面差異、縮排、藍色邊框、箭頭圖示等視覺線索達成）。

## Assumptions

- 假設使用者帳號系統（含註冊、登入、使用者隔離）已存在於前序功能中（參考 specs/001-user-permissions），本規格直接依賴其使用者識別。
- 假設交易實體（Transaction）已存在或將由 specs/002-transactions-accounts 提供，且每筆交易會以外鍵指向 Category；本規格的刪除規則依賴此關聯能被即時查詢。
- 假設「儀表板」「報表」屬於後續或既有功能，本規格僅要求其在使用分類顏色時讀取本系統的顏色欄位，不規範儀表板／報表本身的版面。
- 假設預設顏色由系統內部定義並維持穩定；使用者可改色但系統不主動變更已存在分類的顏色。
- 假設 CSV 匯入／匯出由獨立規格處理，不在本範圍內；本規格只要求模型能容納未來匯入的分類資料（即現有欄位足夠）。
- 假設前端使用單一語系（繁體中文）顯示預設分類名稱；國際化（多語系預設分類名稱）為未來工作，不在本範圍內。
- 假設「登入時補建預設分類」的觸發點為使用者每次登入時的伺服端會話建立流程；補建以 user_id 為粒度同步執行（FR-010a，P95 ≤ 200 ms），失敗時降級為下次重試（FR-010b）。
