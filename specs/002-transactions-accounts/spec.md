# 功能規格：交易與帳戶（Transactions & Accounts）

**Feature Branch**：`002-transactions-accounts`
**Created**：2026-04-24
**Status**：Draft
**Input**：使用者描述：「交易與帳戶：最低摩擦記帳 + 帳戶容器 + 三類交易 + 轉帳對稱 + 外幣自動匯率 + 批次操作 + 篩選分頁 + 電子發票 QR 掃描」

## Clarifications

### Session 2026-04-24

- Q: 帳戶名稱於同一使用者範圍內是否需唯一？ → A: 允許同名（不做唯一性檢查）
- Q: 編輯帳戶幣別對既有交易的影響？ → A: 帳戶已存在任一筆交易時禁止變更幣別（UI 鎖定）
- Q: 交易／帳戶刪除採硬刪除或軟刪除？ → A: 硬刪除（直接從 DB 移除，不設 `deleted_at`、不提供回收桶）
- Q: 批次操作單次最大選取筆數上限？ → A: 500 筆（前端阻擋超過上限的勾選，並於批次操作列顯示剩餘額度）
- Q: 信用卡海外刷卡手續費率儲存位置？ → A: 每張信用卡帳戶各自儲存 `overseas_fee_rate`（預設 1.5%，帳戶編輯頁可調整），交易建立時仍可手動覆寫當筆費率與金額
- Q: 未來日期交易是否計入帳戶餘額與儀表板總資產？ → A: 不計入（餘額 / 總資產 / 所有統計一律 `WHERE date <= CURRENT_DATE`；未來交易僅於列表「未來」分區呈現）
- Q: `fx_fee` 欄位儲存的幣別？ → A: 固定以 TWD 儲存；公式 `twd_amount = amount × fx_rate + fx_fee`（fx_fee 無幣別換算）
- Q: 帳戶 `exclude_from_total` 對統計報表的作用範圍？ → A: 僅影響儀表板「總資產卡」；月支出／分類統計／預算進度一律照算（與交易層級 `exclude_from_stats` 兩軸獨立）
- Q: 併發編輯衝突處理 Policy？ → A: 樂觀鎖（PATCH 須帶 `expected_updated_at`，後端比對不符回 409 Conflict；UI 提示「此筆已被其他裝置修改，請重新整理」）
- Q: 交易列表「自訂每頁筆數」的硬上限？ → A: 500 筆（與批次操作上限對齊；超過前端阻擋、後端回 `400 Bad Request`）
- Q: 跨幣別轉帳如何處理？ → A: 禁止（UI 檢查來源/目標帳戶 currency 必須相同，不同則送出 disabled；後端以 422 拒絕；使用者需改用「支出 + 收入」兩筆替代）
- Q: 金額與匯率的數值精度？ → A: 金額（amount / twd_amount / fx_fee / initial_balance）以幣別最小單位整數儲存（TWD=元、USD=分、JPY=円）；fx_rate 以 decimal 字串（`TEXT`）儲存，後端以 decimal 函式庫計算，避免浮點誤差
- Q: 支援幣別白名單？ → A: Backend 完全開放（接受任意 ISO 4217 三字母代碼，匯率 API 失敗時走 FR-024 fallback）；UI 下拉預設只顯示每位使用者於設定頁自行 pin 的「常用幣別清單」，並保留「新增其他幣別」入口可輸入任意 ISO 4217 代碼
- Q: 時區：`CURRENT_DATE` 的界定？ → A: 固定以 `Asia/Taipei`（UTC+8）為系統時區計算「今天」；DB 內部仍以 UTC 儲存，比對時先轉 Asia/Taipei 再取 date 部分
- Q: API 授權（AuthZ）顯式規範？ → A: 新增 FR 要求所有 `/accounts`、`/transactions`、`/transfers` endpoint 須驗證 session user 對應資源 `user_id`，不符一律回 `404 Not Found`（不洩漏資源存在性）
- Q: 批次操作的結果處理語意？ → A: Atomic 全有全無（整批包一個 DB transaction；任一筆失敗整批 rollback，API 回 `409/422` 並附首筆失敗原因；前端顯示「本次批次操作未生效」）
- Q: 關鍵字搜尋行為？ → A: 模糊 `%keyword%` + case-insensitive + 輸入前後 trim；僅匹配備註欄 `note`（不跨搜分類名稱）
- Q: 交易列表排序選項？ → A: 全欄位排序（`date` / `amount` / `account` / `category` / `type` 皆可 ASC/DESC，共 10 組）；預設 `date DESC`；排序狀態納入 URL query（`sort=<field>_<asc|desc>`）
- Q: 交易日期可接受範圍？ → A: 不設上下限（任何合法 ISO 8601 `DATE` 格式皆可，倚賴使用者自律；後端只驗證格式正確，不檢查範圍）
- Q: 交易匯入規模（批次建立 endpoint）？ → A: 本規格只提供單筆 `POST /transactions`（1 筆/請求）；批次建立交由後續 CSV 匯入 feature 自行設計專屬 endpoint，不在 002 範圍

## 使用情境與測試 *(mandatory)*

### User Story 1 — 建立帳戶並記下第一筆收入或支出（Priority: P1）🎯 MVP

新使用者完成註冊後，儀表板上已有一個自動建立的「現金」帳戶；他在交易頁
新增一筆 `餐飲 → 午餐 / $120 / 今天 / 現金帳戶` 的支出後，帳戶餘額由 $0
變成 -$120、該筆交易出現在列表最上方。若使用者另有「台新銀行」薪資帳戶，
可在「帳戶管理」建立，填入名稱、初始餘額、圖示、類別（銀行）、幣別（TWD），
之後即可以此帳戶為資金來源記帳。

**Why this priority**：「記一筆 → 看餘額」是整個記帳 App 的最短閉環；若 US1
不成立，其他一切功能都是空中樓閣。

**Independent Test**：全新使用者註冊後，(1) 自動擁有「現金」帳戶、(2) 能新
增第二個銀行帳戶、(3) 以任一帳戶記一筆支出、(4) 兩個帳戶的餘額都與「初始
餘額 + 收入 - 支出」的即時計算一致。

**Acceptance Scenarios**：

1. **Given** 新使用者完成註冊，**When** 開啟儀表板，**Then** 帳戶列表顯示一個名為「現金」的預設帳戶，初始餘額 0、圖示 `fa-wallet`、類別「現金」、幣別 TWD。
2. **Given** 使用者在「帳戶管理」按「新增帳戶」，**When** 填入「台新銀行 / $50,000 / `fa-university` / 銀行 / TWD」並送出，**Then** 帳戶列表新增一列顯示 $50,000。
3. **Given** 使用者在交易頁按「新增交易」，**When** 選「支出 / 餐飲 → 午餐 / $120 / 今天 / 現金帳戶 / 備註『麥當勞』」並送出，**Then** 交易列表新增一筆 `2026-04-24 支出 -$120 現金 餐飲>午餐 麥當勞`、現金帳戶餘額顯示 -$120。
4. **Given** 使用者於金額欄位輸入 `0` 或 `-100`，**When** 嘗試送出，**Then** 前後端都拒絕，訊息明確提示「金額必須大於 0」。
5. **Given** 使用者於日期欄位選未來日期（例：下個月 5 日），**When** 送出，**Then** 交易成功建立，列表該列附「未來」標籤且與今日／過往交易於視覺上分區；**該筆交易不影響當前帳戶餘額與儀表板總資產（直到日期到達）。**

### User Story 2 — 查看交易列表與帳戶即時餘額（Priority: P1）🎯 MVP

使用者開啟交易頁看到全部交易依日期新到舊排列；頂端可用日期範圍、類型、
分類、帳戶、關鍵字（備註）做篩選，換頁時可選每頁 10／20／50／100 或自訂
任意筆數；回到儀表板可見總資產卡顯示「所有計入總資產的帳戶」之 TWD 等
值加總，明顯區分「已排除」帳戶。

**Why this priority**：可看見的結果是記帳成就感的根本來源，沒有可視性就沒
有持續使用的動機。

**Independent Test**：建立 60 筆測試資料後，(1) 每頁 20 時正確顯示分頁器、
(2) 篩選「支出」+「餐飲」類時僅剩對應筆數、(3) 帳戶切換時列表即時更新、
(4) 儀表板總資產等於所有計入帳戶餘額之加總。

**Acceptance Scenarios**：

1. **Given** 已有 60 筆交易，**When** 開啟交易頁，**Then** 預設每頁 20、共 3 頁、依 `date DESC` 排序；類型欄以色彩標籤呈現：支出紅、收入綠、轉帳藍。
2. **Given** 交易頁已載入，**When** 於頂端搜尋欄輸入「咖啡」，**Then** 列表僅保留備註含「咖啡」的交易。
3. **Given** 使用者於篩選列選「類型 = 轉帳」，**When** 套用，**Then** 列表僅顯示 `transfer_out` 與 `transfer_in` 兩種方向，且成對互相標示。
4. **Given** 使用者於分頁器「每頁筆數」選「自訂」並輸入 37，**When** 送出，**Then** 列表改為每頁 37 筆並重新計算總頁數。
5. **Given** 有 5 個帳戶其中 2 個標記為不計入總資產，**When** 儀表板載入，**Then** 總資產卡僅加總 3 個計入帳戶；2 個排除帳戶有明顯標籤（例如灰色虛線外框 + 「已排除」徽章）。

### User Story 3 — 以轉帳記錄信用卡消費與繳款（Priority: P2）

使用者刷信用卡買 $3,000 的家電，他在信用卡帳戶上記一筆「支出 $3,000」。
月底從銀行帳戶繳款 $3,000，此時他新增「轉帳：銀行帳戶 → 信用卡 $3,000」。
轉帳產生對稱的 `transfer_out` + `transfer_in` 兩列互以 `linked_id` 關聯，繳款
金額中和信用卡帳戶餘額的負值；統計報表只認列消費那筆「支出」，繳款不重
複計入支出。若使用者刪除任一半，另一半自動同步刪除。

**Why this priority**：若不支援轉帳，信用卡帳務與跨帳戶資金流動就會錯誤
地重複計入或漏計；但 US1 已能完成單一帳戶記帳，因此優先級為 P2。

**Independent Test**：(1) 於信用卡帳戶記 $3,000 支出 → 餘額 -$3,000、月
支出統計 +$3,000、(2) 新增銀行 → 信用卡 $3,000 轉帳 → 信用卡餘額 $0、銀行餘
額 -$3,000、月支出統計**不增加**、(3) 刪除 `transfer_in` 那一半 → `transfer_out`
那一半也被刪、兩個帳戶的餘額同步回復。

**Acceptance Scenarios**：

1. **Given** 使用者於交易 Modal 選「轉帳」，**When** 填入「來源：銀行帳戶 / 目標：信用卡 / 金額 $3,000 / 日期：今天」並送出，**Then** 系統在 `transactions` 新增兩列：一列 `transfer_out`（銀行，-$3,000）、一列 `transfer_in`（信用卡，+$3,000），兩者共享 `linked_id`。
2. **Given** 上述轉帳已存在，**When** 使用者於列表刪除 `transfer_in` 那一筆，**Then** 二次確認 Modal 明確告知「這是一組轉帳，對應的 transfer_out 將一併刪除」；確認後兩筆一起消失。
3. **Given** 同上，**When** 產月支出統計，**Then** 該筆 $3,000 僅於「支出類」出現一次（原消費那筆），轉帳兩列均不計入支出／收入統計。
4. **Given** 使用者在信用卡帳戶頁看到餘額 -$3,000、且該卡分組下出現「還款」捷徑，**When** 點擊捷徑，**Then** 轉帳 Modal 預填「目標 = 此信用卡 / 金額 = 目前未還款金額 $3,000」，僅需選擇來源銀行即可送出。

### User Story 4 — 批次處理多筆交易（Priority: P2）

使用者於交易列表勾選 40 筆舊交易，點出現的紫色批次操作列執行「批次變更分類」
改到「娛樂→電影」、或「批次變更日期」統一設為月初、或「批次刪除」清理試
錄資料。批次變更分類的選單使用自訂元件顯示色點與分區結構，避免原生 `<optgroup>` 在部分
瀏覽器渲染不全。批次刪除遇到轉帳時會提示「對應另一半也會被刪除」。

**Why this priority**：一次處理一批交易對重度使用者效率提升顯著；但 US1／US2
已能逐筆完成同樣的修改，故列為 P2。

**Independent Test**：建 50 筆交易 →（1）全選後出現批次操作列、（2）執行批次刪
除含 10 筆轉帳 → 轉帳對應的另一半也消失、（3）勾選 20 筆執行「批次變更分類」→
所選 20 筆全部改為新分類、其他未勾選者不受影響。

**Acceptance Scenarios**：

1. **Given** 列表載入 50 筆交易，**When** 點表頭全選 checkbox，**Then** 所有列同時勾選、表頭 checkbox 為完整勾選狀態、出現紫色批次操作列顯示「已選 50」。
2. **Given** 已勾選 20 筆，**When** 再點其中 5 筆取消勾選，**Then** 表頭 checkbox 變為半選狀態（indeterminate），批次操作列顯示「已選 15」。
3. **Given** 使用者於批次操作列選「批次變更分類」，**When** 彈出自訂下拉選單（含色點 + 父子階層分區），選「娛樂 → 電影」套用，**Then** 選中的 N 筆全部更新分類；列表該 N 列之「分類」欄立即重繪；其他交易不受影響。
4. **Given** 已勾選 3 筆（含 1 筆轉帳），**When** 點「批次刪除」，**Then** 二次確認 Modal 告知「共 3 筆含 1 組轉帳（對應另一半將一併刪除，實際將刪除 4 筆）」；確認後 4 筆一起消失。

### User Story 5 — 外幣交易與自動匯率（Priority: P2）

使用者在東京出差刷卡買了 ¥10,000 的餐點，他新增「支出」並把幣別下拉切成
JPY，系統自動抓到當下即時匯率 0.2103，TWD 等值 $2,103 自動填入；因為選
的是信用卡帳戶，系統再自動加 1.5% 的海外刷卡手續費 $31，讓最終記錄為 JPY
¥10,000 + 手續費 TWD $31、入帳 TWD $2,134。送出後該匯率與手續費率固
化於該筆交易，就算明日匯率變動，這筆歷史紀錄與月報表的 TWD 等值不會飄
移。匯率查詢是跨使用者共用快取（同幣別 5 分鐘內請求去重 + 伺服端 30 分鐘快
取）。

**Why this priority**：對部分使用者是剛需（常出國／跨境），但對本地使用者
零影響；若未實作，外幣交易只能用使用者手動填匯率，體驗差但不阻擋。

**Independent Test**：(1) 於交易 Modal 把幣別由 TWD 改為 JPY → 同頁面 < 2 秒
內自動填入當下匯率、TWD 等值動態計算顯示；(2) 選信用卡帳戶時自動追加
1.5% 手續費欄位，可手動調整；(3) 儲存交易後於 `exchange_rates` 記錄當下匯率
並固化至該筆交易的 `fx_rate`；(4) 日後修改系統當前匯率，原交易的 TWD 等值不變。

**Acceptance Scenarios**：

1. **Given** 交易 Modal 已開啟、幣別預設 TWD、金額欄為空，**When** 使用者把幣別下拉改為 JPY，**Then** 系統呼叫匯率 API 取得 JPY→TWD 匯率並於 2 秒內填入「匯率」欄位；TWD 等值欄位隨金額輸入即時計算。
2. **Given** 幣別 = JPY、帳戶類別 = 信用卡，**When** 使用者填入金額 ¥10,000，**Then** UI 自動顯示「海外刷卡手續費 1.5%（可調整）= TWD $31」；最終 TWD 入帳 = 原幣等值 + 手續費。
3. **Given** 匯率 API 逾時或失敗，**When** 使用者儲存，**Then** 系統以最近一次成功快取值（若 30 分鐘內）作為預設；若無可用快取則提示「匯率暫不可用，請手動輸入」。
4. **Given** 某使用者剛查過 JPY 匯率（5 分鐘內），**When** 另一使用者同時間選 JPY，**Then** 系統不重複呼叫外部 API（命中 in-flight dedup 或 30 分鐘快取），回應時間 < 100ms。
5. **Given** 交易已儲存時匯率 0.2103，**When** 7 天後匯率變為 0.2050，**Then** 該筆交易的 `fx_rate` 欄位仍為 0.2103、月報表 TWD 等值不變。

### User Story 6 — 電子發票 QR 掃描自動填單（Priority: P3）

使用者在便利商店結帳後拿到發票，他在手機上開啟交易 Modal，點「掃描發票」
按鈕，App 呼叫相機對準 QRCode；系統解析財政部電子發票**左碼**（含發票
基本資訊：發票字軌／日期／隨機碼／銷售額／總計／買賣方統編），自動填入
金額、日期、賣方統編字串（寫入備註欄，例如「賣方統編：12345678」，
使用者可手動改寫為店家名稱）三個欄位；使用者再補選分類與帳戶即可送出。
若相機不可用，可改為上傳已拍攝的圖片。**右碼（品項明細）解析、賣方統編
反查店家名稱表為 Phase 2 範圍外**，本階段不實作。

**Why this priority**：大幅降低行動端記帳摩擦，但桌面版或非台灣使用者無
法使用；US1 已能滿足記帳本身。

**Independent Test**：(1) 於支援相機的行動瀏覽器開 Modal → 按掃描 → 對準
發票 QR → 3 秒內金額／日期／賣方統編字串自動填入；(2) 於桌面瀏覽器不
支援相機時提供「上傳圖片」fallback；(3) QR 解析失敗時明確告知，不破壞
已填欄位。

**Acceptance Scenarios**：

1. **Given** 交易 Modal 已開啟、裝置支援 `MediaDevices.getUserMedia`，**When** 使用者點「掃描發票」，**Then** 取得使用者同意後啟動後鏡頭預覽。
2. **Given** 鏡頭對準財政部電子發票 QRCode，**When** 成功解析左碼（發票基本資訊區段），**Then** 金額、日期、賣方統編字串（備註欄填入「賣方統編：{8 數字}」）自動填入表單；使用者確認、（如需要）手動改寫備註為店家名稱，並補選分類 + 帳戶後即可送出。**右碼（品項明細）本階段不解析。**
3. **Given** 裝置不支援相機或使用者拒絕授權，**When** 點「掃描發票」，**Then** 改顯示「上傳圖片」介面；上傳後以同樣解析器處理。
4. **Given** 上傳的圖片不含有效 QRCode，**When** 解析完成，**Then** 顯示錯誤訊息「無法解析電子發票 QRCode」；原本已手動填入的欄位不被清空。

### 邊界案例（Edge Cases）

- 帳戶已被交易引用時嘗試刪除：拒絕並提示「請先處理該帳戶上的 N 筆交易（可批次移到其他帳戶或刪除）」。
- 刪除交易時，若為轉帳的任一半：二次確認 Modal 明確提示另一半會被刪除。
- 批次刪除混合含轉帳與單邊交易：確認 Modal 明確顯示實際刪除筆數（勾選筆數 + 被連動的 `linked` 另一半）。
- 信用卡帳戶餘額長期為負是合法狀態（未結清消費）；使用者修改金額為 0 或負數時仍禁止。
- 未來日期的交易被納入「未來」標籤分區，**不影響當前帳戶餘額、總資產卡、月支出／分類統計／預算進度**；需待日期到達（`date <= CURRENT_DATE`）後才於次一次頁面載入反映至所有數字。
- 匯率 API 全失敗 + 無快取：允許使用者手動輸入，不阻擋儲存。
- 幣別從 TWD 改為 JPY 後又改回 TWD：匯率欄位自動清為 1（或隱藏）、手續費欄消失。
- 轉帳 Modal 使用者選擇來源 TWD 帳戶、目標 JPY 帳戶：送出按鈕 disabled 並顯示「跨幣別請分開記一筆支出 + 一筆收入」；後端遇到繞過前端檢查的跨幣別請求回 `422 Unprocessable Entity`。
- 「不計入統計」的交易仍影響帳戶餘額，但不影響儀表板／分類統計／預算進度。
- E-invoice QR 解析出的金額為 0（退換貨發票）：仍填入但前端標示警告，使用者需手動調整。
- 使用者同時在兩個裝置開同一張交易編輯：採樂觀鎖（`expected_updated_at` 比對），後送者收到 `409 Conflict` 並被提示重新整理；不保留版本歷史（不儲存舊值，僅以 `updated_at` 判定衝突）。

## 需求 *(mandatory)*

### Functional Requirements

#### 帳戶

- **FR-001**：系統必須允許使用者新增帳戶，欄位包含：名稱（必填，**同使用者範圍內允許同名**，不做唯一性檢查）、初始餘額（數字，允許 0 或負值）、圖示（FontAwesome class）、帳戶類別（`bank` / `credit_card` / `cash` / `virtual_wallet` 其一）、幣別（ISO 4217 三字母代碼，預設 TWD）、是否計入總資產（預設是）。**當類別為 `credit_card` 時必須額外提供 `overseas_fee_rate`（百分比，預設 1.5，允許 0～10，可於帳戶編輯頁調整）。**
- **FR-002**：系統必須於使用者註冊成功時自動建立一個預設「現金」帳戶：名稱「現金」、初始餘額 0、圖示 `fa-wallet`、類別 `cash`、幣別 TWD、計入總資產。
- **FR-003**：系統必須提供帳戶管理頁，以帳戶類別作為 tab 切換分頁並分組顯示；信用卡類別帳戶必須額外支援「銀行分組」（依連結的還款來源銀行帳戶分組）與「一鍵還款」捷徑，捷徑會開啟預填「目標 = 此信用卡 / 金額 = 目前未還款金額」的轉帳 Modal。
- **FR-004**：系統必須提供「是否計入總資產」開關（布林，`exclude_from_total`），排除的帳戶在帳戶列表顯示「已排除」徽章，並在儀表板總資產卡註明「已排除 N 個帳戶」。**本開關僅影響總資產卡；該帳戶上的交易在月支出／分類統計／預算進度等所有其他統計報表中一律照常計入。若使用者需從報表排除特定交易，請改用交易層級的 `exclude_from_stats`（見 FR-017）。**
- **FR-005**：系統必須允許編輯帳戶欄位（名稱、圖示、類別、初始餘額、是否計入總資產）；編輯後的餘額依新初始餘額即時重算。**幣別欄位僅於帳戶尚無任何交易時允許變更；一旦該帳戶存在至少 1 筆 `transactions` 引用（`account_id` 或 `to_account_id`），幣別欄位必須於 UI 鎖定且後端拒絕變更，錯誤訊息須提示「此帳戶已有交易紀錄，無法變更幣別；如需不同幣別請新增帳戶」。**
- **FR-006**：系統必須在刪除帳戶前檢查 `transactions.account_id`／`transactions.to_account_id`（轉帳目標）是否仍有引用；若有則拒絕刪除，錯誤訊息明確告知引用筆數與可行解法（移到其他帳戶或先刪除那些交易）。
- **FR-007**：系統必須即時由交易紀錄計算每個帳戶的當下餘額，公式為 `初始餘額 + Σ(type=income 金額) − Σ(type=expense 金額) + Σ(type=transfer_in 金額) − Σ(type=transfer_out 金額)`，**聚合範圍僅包含 `date <= CURRENT_DATE` 的交易（未來日期交易不納入餘額計算）**；不另外儲存「目前餘額」欄位。儀表板總資產卡與所有統計報表（月支出、分類統計、預算進度）一律套用相同的 `date <= CURRENT_DATE` 過濾。
- **FR-007a**：所有「今天／未來」的判定（FR-007、FR-013、未來分區、報表過濾）必須以 **`Asia/Taipei` (UTC+8) 固定時區**計算 `CURRENT_DATE`；伺服器內部時間戳記（如 `created_at`、`updated_at`、`exchange_rates.fetched_at`）仍以 UTC 儲存，比對日期時先以 `Asia/Taipei` 轉換後取 date 部分。`transactions.date` 欄位只存 `DATE`（無時間成分），代表使用者當地（台灣時區）的自然日。

#### 交易

- **FR-010**：交易必須有三種類型：`income`（收入）、`expense`（支出）、`transfer_out` / `transfer_in`（轉帳，成對存在）。
- **FR-011**：收入與支出交易必須具備以下欄位：金額（> 0，不接受 0 或負值）、日期（允許未來日期）、分類（必填）、帳戶（必填）、備註（選填，上限 200 字元）。
- **FR-012**：分類下拉選單必須以 `<optgroup>` 或等效 UI 顯示父分類 → 子分類階層；批次變更分類的下拉選單必須採自訂元件（含色點 + 分區結構），以規避瀏覽器對多層 `<optgroup>` 的渲染缺陷。
- **FR-013**：交易日期允許未來日期以供預先規劃；列表必須將未來日期交易以「未來」標籤與歷史交易分區顯示。**日期欄位不設上下限**，任何合法 ISO 8601 `DATE`（`YYYY-MM-DD`）格式皆可接受；後端僅驗證格式正確性（拒絕非 `DATE` 格式字串），不檢查日期落點範圍。
- **FR-014**：編輯交易時僅記錄最後修改時間（`updated_at`），不保留版本歷史；刪除必須出現二次確認 Modal。**刪除採硬刪除（`DELETE FROM transactions`），不使用 `deleted_at` 欄位、不提供回收桶；帳戶刪除同此原則。**
- **FR-014a**：交易與帳戶的 PATCH／DELETE API 必須採**樂觀鎖**：請求 body 或 header 須帶 `expected_updated_at`（ISO 8601 UTC 或 epoch ms），後端於更新前比對資料庫現值；不符即回 `409 Conflict`，UI 須提示「此筆已被其他裝置修改，請重新整理後再操作」並禁止強制覆寫。本規則同時適用於 `transactions` 與 `accounts` 兩表。
- **FR-015**：轉帳必須以一組 `transfer_out` + `transfer_in` 兩筆交易組成，透過 `linked_id` 互相關聯；刪除任一半時系統必須自動同步刪除另一半。**轉帳僅支援同幣別：來源帳戶與目標帳戶的 `currency` 必須相同；UI 在兩者不同時送出按鈕 disabled 並提示「跨幣別請分開記一筆支出 + 一筆收入」，後端收到跨幣別轉帳請求須以 `422 Unprocessable Entity` 拒絕。**
- **FR-016**：統計報表（儀表板、分類統計、預算進度）僅認列 `income` 與 `expense`；轉帳兩筆均不計入支出／收入統計。
- **FR-017**：單筆交易必須可標記為「不計入統計」（`exclude_from_stats` 布林）；被標記的交易仍影響帳戶餘額，但不影響儀表板、分類統計、預算進度。
- **FR-018**：交易列表的類型欄位必須以顏色標籤呈現：支出紅、收入綠、轉帳藍。

#### 外幣與匯率

- **FR-020**：交易建立時選擇非 TWD 幣別，系統必須自動呼叫匯率 API 取得當下該幣別對 TWD 的即時匯率並填入「匯率」欄位；TWD 等值欄位隨金額與匯率即時計算。
- **FR-020a**：系統必須允許任意 ISO 4217 三字母代碼作為帳戶／交易幣別（backend 不做白名單限制，匯率 API 失敗時走 FR-024 fallback）；但 UI 下拉選單預設**僅顯示當前使用者於設定頁 pin 的「常用幣別清單」**（預設至少包含 TWD），下拉底部必須提供「新增其他幣別」入口，使用者可輸入任意 ISO 4217 代碼加入 pin 清單後使用。常用幣別清單於使用者偏好 (`user_settings.pinned_currencies`) 儲存，為跨裝置同步的使用者層級偏好。
- **FR-021**：若交易所屬帳戶類別為信用卡且幣別非 TWD，系統必須額外顯示「海外刷卡手續費」欄位，**預設費率取自該信用卡帳戶的 `overseas_fee_rate`（若未設定則 fallback 1.5%）**；使用者可於當筆交易手動覆寫費率與金額（覆寫結果僅作用於該筆交易，不回寫帳戶欄位）。**`fx_fee` 欄位恆以 TWD 儲存**，最終 TWD 入帳金額 `twd_amount = amount × fx_rate + fx_fee`（三個量皆無幣別換算 ambiguity）。
- **FR-022**：交易儲存時，採用的 `fx_rate` 與 `fx_fee` 必須與交易綁定持久化；日後系統匯率變動不影響已儲存交易的歷史 TWD 等值。
- **FR-022a**：所有金額欄位（`amount`、`twd_amount`、`fx_fee`、`accounts.initial_balance`）必須以幣別最小單位之整數儲存（TWD：元；USD/EUR：分；JPY：円），DB 型別 `INTEGER`。`fx_rate` 必須以 decimal 字串儲存（DB 型別 `TEXT`，建議精度至少 8 位小數）；後端所有匯率與金額運算須採 decimal 函式庫（如 Python `decimal.Decimal`、JS `decimal.js`），禁用原生 float/Number 直接相乘，以避免浮點 rounding 誤差。前端 display 時再依幣別定義的小數位格式化呈現（例如 TWD `$1,200`、USD `$12.00`、JPY `¥1,200`）。
- **FR-023**：匯率查詢必須為跨使用者共用快取：同幣別 5 分鐘內以 in-flight request deduplication 合併並發請求；另以伺服器端 30 分鐘快取降低外部 API 呼叫。
- **FR-024**：匯率 API 失敗時，系統必須依序採：(1) 最近一次成功快取（< 30 分鐘）、(2) 使用者手動輸入；若兩者皆無則明確提示「匯率暫不可用，請手動輸入」，不阻擋儲存。

#### 電子發票

- **FR-030**：交易新增 Modal 必須提供「掃描發票 QRCode」按鈕；裝置支援 `MediaDevices.getUserMedia` 時啟動相機預覽，否則改以「上傳圖片」介面 fallback。
- **FR-031**：系統必須解析財政部電子發票**左碼**（含發票基本資訊：發票字軌／日期／隨機碼／銷售額／總計／買方統編／賣方統編），自動填入金額、日期、**賣方統編字串**（備註欄寫入「賣方統編：{8 數字}」格式，使用者可手動改寫為店家名稱）；分類與帳戶需使用者補選。**右碼（品項明細）解析、賣方統編→店家名稱反查表，列為 Phase 2 範圍外**（本階段不實作；理由：避免引入賣方資料庫，見 [research.md](./research.md) §6.3）。
- **FR-032**：QR 解析失敗時必須明確提示「無法解析電子發票 QRCode」，且不清空使用者已手動填入的欄位。

#### 批次操作

- **FR-040**：交易列表必須支援多選 checkbox，包含「全選」、「半選」（indeterminate）、「取消全選」三種表頭狀態。
- **FR-041**：勾選任一筆交易後必須出現批次操作列（紫色視覺強調），顯示已選筆數並提供操作按鈕：批次刪除、批次變更分類、批次變更帳戶、批次變更日期。
- **FR-042**：批次刪除涉及轉帳的任一半時，必須於二次確認 Modal 明確告知「含 N 組轉帳對；實際將刪除 M 筆」。
- **FR-043**：批次變更分類／帳戶／日期必須是單次選單操作（不開多個 Modal）；套用後列表受影響的欄位即時重繪，未勾選的交易不受影響。
- **FR-044**：批次操作（刪除／變更分類／變更帳戶／變更日期）單次最大選取筆數為 **500 筆**；前端在選取數達 500 時禁止繼續勾選並於批次操作列提示「已達單次上限 500，請分批處理」；後端在收到超過 500 筆的請求時須以 `400 Bad Request` 拒絕，避免單次 DB transaction 過久鎖表。
- **FR-045**：批次操作必須以 **atomic 全有全無**語意執行：整批包在單一 DB transaction 內，任一筆違反約束（如樂觀鎖 `expected_updated_at` 不符、帳戶不屬於當前使用者、目標分類不存在）即全部 rollback，不留任何部分生效結果。API 回應以首筆失敗原因為 `error.reason`，狀態碼依失敗類型選用 `409 Conflict`（併發衝突）、`422 Unprocessable Entity`（資料違反約束）或 `404 Not Found`（授權/不存在）；前端必須顯示「本次批次操作未生效，{首筆失敗原因}」，不得呈現「N 筆成功、M 筆失敗」的混合結果。

#### 篩選與分頁

- **FR-050**：交易列表預設以 `date DESC` 排序；頂端篩選列必須支援：日期範圍、類型（`income` / `expense` / `transfer` / 未來交易 / 全部）、分類、帳戶、關鍵字搜尋（匹配備註欄 `note`，採**模糊中綴匹配 `%keyword%` + case-insensitive + 輸入前後 trim 空白**；不跨搜分類名稱或其他欄位）；多條件為 AND 關係。列表表頭必須支援**使用者切換排序欄位**，可選 `date` / `amount` / `account`（依帳戶名稱）/ `category`（依分類名稱）/ `type`，每欄位可獨立切換 ASC / DESC（共 10 組組合）；排序狀態須納入 URL query（格式 `sort=<field>_<asc|desc>`，例：`sort=amount_desc`）以對齊 FR-052 的可還原需求。
- **FR-051**：每頁筆數必須可選 10 / 20 / 50 / 100 或「自訂」；自訂模式允許輸入 1～**500** 的正整數（硬上限 500，與批次操作上限一致）；前端在輸入 > 500 時禁止送出並提示「每頁最多 500 筆」，後端在收到 `page_size > 500` 時回 `400 Bad Request`。
- **FR-052**：篩選與分頁參數必須可反映於 URL（query string 或 hash），以便使用者分享連結或瀏覽器書籤可還原同一視圖。

#### 授權與存取控制

- **FR-060**：所有 `/accounts`、`/transactions`、`/transfers` 及衍生 endpoint（含讀取、建立、更新、刪除、批次操作）必須於授權中介層驗證 request 對應 session user 的 `user_id` 與目標資源 `user_id` 一致；不一致時一律回應 `404 Not Found`（而非 `403 Forbidden`，以避免透過狀態碼差異列舉他人資源 ID）。此規則同時覆蓋單筆查詢、清單過濾（他人 `user_id` 不得出現在回應）、轉帳目標帳戶（`to_account_id` 亦須屬於同一使用者）。此為針對 OWASP API Top 10 A1 (Broken Object Level Authorization / IDOR) 的強制防線，任何 endpoint 新增皆須套用同一中介層。

### Key Entities

- **Account（帳戶）**：資金容器。關鍵屬性：id、name、icon、category（bank / credit_card / cash / virtual_wallet）、currency、initial_balance、exclude_from_total、linked_bank_id（信用卡用；指向還款來源銀行帳戶）、`overseas_fee_rate`（僅 credit_card 類別；預設 1.5，範圍 0~10）、created_at。與交易為一對多關聯（`account_id` + `to_account_id`）。
- **Transaction（交易）**：資金流向事件。關鍵屬性：id、user_id、account_id、to_account_id（轉帳時指向目標帳戶）、type（income / expense / transfer_out / transfer_in）、**amount（原幣金額，`INTEGER` 幣別最小單位）**、currency、**fx_rate（`TEXT` decimal 字串，≥ 8 位小數）**、**fx_fee（TWD 最小單位整數）**、**twd_amount（TWD 最小單位整數，`= amount × fx_rate + fx_fee` 以 decimal 計算後取整）**、category_id、date、note（上限 200 字）、exclude_from_stats、linked_id（轉帳對的另一半）、created_at、updated_at。
- **ExchangeRate（匯率快取）**：跨使用者共用的幣別匯率查詢快取。關鍵屬性：currency（如 JPY）、rate（對 TWD）、fetched_at、source（API 名稱）；5 分鐘 in-flight deduplication + 30 分鐘 server-level 快取。
- **Category（分類）**：交易歸類。於本規格範圍外（屬 001-user-permissions 預設分類已建立），此處僅作為引用。

## 成功標準 *(mandatory)*

### Measurable Outcomes

- **SC-001**：新使用者註冊完成後，第一筆交易（支出）從「開啟交易頁」到「送出後看到列表更新」可在 60 秒內完成（含填寫 5 個欄位）。
- **SC-002**：帳戶餘額必須在任一筆交易新增／編輯／刪除後，於下次帳戶頁載入時立即反映；從使用者按「送出」到餘額更新可見的延遲 P95 < 1 秒（不含網路往返）。
- **SC-003**：外幣交易建立時，從使用者切換幣別到匯率自動填入的延遲 P95 < 2 秒；同幣別 5 分鐘內的第二次查詢延遲 < 100 ms（快取命中）。
- **SC-004**：轉帳交易刪除任一半時，另一半必須於同次請求內被一併刪除；以 100 組轉帳壓測，不允許出現任何「孤兒」transfer_out 或 transfer_in 殘留。
- **SC-005**：批次操作（刪除 / 變更分類 / 變更帳戶 / 變更日期）在單筆 100 筆交易勾選的情境下，從點擊到列表完成重繪的延遲 P95 < 3 秒。
- **SC-006**：儲存後的交易 `fx_rate` 不隨系統當前匯率變動；以「30 天間匯率改變 5 次」模擬，原交易的 TWD 等值維持不變。
- **SC-007**：篩選列與分頁參數必須可完整還原：複製交易頁 URL 到另一個瀏覽器 session 後，相同篩選／分頁狀態必須一致重現。
- **SC-008**：帳戶已被交易引用時嘗試刪除，100% 會被拒絕；錯誤訊息明確告知引用筆數。

## 假設 *(Assumptions)*

- 使用者帳戶權限與基本登入功能由 001-user-permissions 提供，本規格僅依賴其建立的 user 身分；預設分類（餐飲／交通／購物…）由 001 的 `createDefaultsForUser()` 建立。
- 匯率 API 以 exchangerate-api.com 免費方案為預設來源（環境變數 `EXCHANGE_RATE_API_KEY=free`）；若使用者升級付費 key，快取與 dedup 行為不變。
- 電子發票解析僅支援財政部電子發票的標準左碼（發票基本資訊區段）；右碼（品項明細）與賣方統編→店家名稱反查表為 Phase 2 範圍外；不支援超商整合發票等變形版本（亦 Phase 2 範圍外）。
- 所有統計與儀表板一律以 TWD 為基準，不做多幣別報表切換；此為刻意縮小範圍的設計決策（見「不做什麼」）。
- 不做自動銀行對帳／券商匯入；交易來源限於：手動新增、固定收支排程、CSV 匯入（CSV 匯入屬另一功能，本規格不涵蓋）。
- 不保留交易修改歷史；僅記 `updated_at`。此為刻意取捨（降低儲存成本 + 簡化 UI）。
- 未來日期交易於排程時（固定收支觸發）才實際計入報表；本規格不涵蓋固定收支邏輯（屬另一功能）。
- 資料庫為單節點 SQLite（sql.js），同使用者的交易操作假設序列化執行，不處理多客戶端並發寫入衝突。

## 不做什麼（Non-Goals）

- **不做自動銀行對帳／券商 API 匯入**：交易來源限手動、固定收支、CSV 匯入。
- **不做多幣別報表切換**：儀表板／月報／分類統計一律以 TWD 等值為準；個別交易的原幣金額僅於列表顯示。
- **不保留交易修改歷史**：僅記 `updated_at`；誤改必須由使用者手動復原。
- **不支援跨使用者共用帳戶／交易**：所有資料以 `user_id` 為隔離邊界。
- **不提供批次建立交易 endpoint**：`POST /transactions` 單次僅接受 1 筆；CSV 匯入、固定收支排程產生多筆交易等批次建立需求，由對應的後續 feature（CSV 匯入、固定收支）自行設計專屬 endpoint，不在 002 範圍。本規格的「批次操作」(FR-040~FR-045) 僅指既有交易的批次更新／刪除，不含批次建立。
