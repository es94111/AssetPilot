**Created**: 2026-04-27
**Status**: Draft
**Input**: User description: "2.7 資料匯出匯入 — 交易／分類／股票 CSV 匯出匯入、Formula Injection 防護、轉帳自動配對、SQLite 整檔備份（管理員）、全球即時匯率自動更新（exchangerate-api.com）、API 使用與授權資訊頁；不做 Excel/OFX/QIF/雲端同步"

## Clarifications

### Session 2026-04-27

- Q: 股票交易與股利匯入時，以哪些欄位組成「視為重複而略過」的唯一鍵？ → A: **股票交易**：日期 + 代號 + 類型 + 股數 + 成交價 + 帳戶 完全相同 → 略過；**股利**：日期 + 代號 + 現金股利 + 股票股利 完全相同 → 略過（與 FR-014 一般交易精神一致，但欄位調整為 CSV 實際存在欄位）
- Q: 當同日期同金額有多筆轉出與轉入候選時，配對演算法應如何決定誰配誰？ → A: 帳戶感知 + 順序兜底 — 若兩邊都還有未配對候選，以 CSV 中出現順序兩兩配對寫入相同 `linked_id`；若僅一邊有剩餘候選（多出的轉出或多出的轉入），多出的那筆寫為單向交易並列入「警告」清單供人工後處理
- Q: 「自動建立缺項 + 匯入交易」過程中發生 partial failure 時的原子性？ → A: 全交易性 — 自動建立的分類／帳戶 + 所有交易包在單一 DB transaction，任一失敗則整批 rollback（含已建立的分類／帳戶），使用者重試時是乾淨白板；UI 顯示明確錯誤訊息與失敗位置
- Q: 管理員下載未加密 SQLite 備份檔時，是否強制安全提示？ → A: 下載前一次性確認彈窗 — 提示「此檔案包含明文敏感資料（含使用者密碼雜湊、session token 等），請妥善保管於受控環境，勿透過 email／雲端硬碟轉存」；管理員點「確認下載」才觸發下載；不引入加密／密碼包裝（避免密碼遺失即無法還原的災難）
- Q: 股利 CSV 應如何補上「帳戶」欄位以符合 006 規格的「現金股利 > 0 時必填入款帳戶」要求？ → A: 新增「帳戶」欄位於備註前，CSV 欄位變為：日期、股票代號、股票名稱、現金股利、股票股利、**帳戶**、備註；匯入時若現金股利 > 0 而帳戶欄位空白則該行歸為「錯誤」；純股票股利（現金股利 = 0）時帳戶欄位可留空
- Q: 所有 CSV 中「日期」欄位的接受／匯出格式為何？ → A: 嚴格 ISO 8601（`YYYY-MM-DD`）— 匯出固定採此格式；匯入僅接受 `YYYY-MM-DD`，其他格式（`YYYY/M/D`、`MM/DD/YYYY`、民國紀年等）一律歸為「錯誤」並附原因「日期格式必須為 YYYY-MM-DD」
- Q: 分類 CSV「顏色」欄位的接受／匯出格式為何？ → A: 嚴格 hex 6 碼 — 匯出固定 `#RRGGBB`（六字元 hex，大小寫不限）；匯入僅接受正規式 `^#[0-9A-Fa-f]{6}$`，3 碼短寫、CSS 顏色名、rgb() 等格式一律拒絕並歸為「錯誤」
- Q: 同使用者並行匯入請求的處理策略為何？ → A: 互斥鎖（排他）— 同使用者同時最多 1 個匯入請求進行中；後端以「使用者匯入鎖」拒絕第二個請求並回 409；UI 顯示「您已有匯入進行中，請稍候完成後再試」；前端同時禁用上傳按鈕避免誤點
- Q: 管理員「還原資料庫」過程中發生 partial failure 時的處置策略？ → A: 自動回滾 — 偵測覆寫失敗時，系統 MUST 以稍早自動寫入的 `backups/before-restore-{timestamp}.db` 覆蓋回主資料庫，UI 提示「還原失敗，已自動回復至還原前狀態，請檢查 server 日誌」；該事件 MUST 寫入 server 日誌（含失敗原因與時間戳）以利後續排查
- Q: 匯出／匯入／備份／還原是否需要 audit log？ → A: 全功能 audit log — 所有匯出（交易／分類／股票／股利／備份）與所有匯入（交易／分類／股票／股利／還原）均寫入統一 `data_operation_audit_log` 表；欄位含 user_id、role、action、ip_address、user_agent、timestamp、result（success／failed／rolled_back）、metadata（檔案大小、筆數、失敗原因等）；管理員專屬操作（下載備份／還原資料庫）額外標記 `is_admin_operation = true`
- Q: 純股票股利（現金股利 = 0、股票股利 > 0、CSV 帳戶留空）匯入時，FR-023 合成 $0 買進交易要綁哪個帳戶？ → A: 沿用該 symbol 最近一筆買進交易的 account；若該股票尚無任何既有買進紀錄（首次匯入），fallback 至使用者唯一存在的證券型帳戶；若使用者有多個證券帳戶且該股票無既有歷史，則該行歸為「錯誤」並提示「純股票股利合成交易無法判定所屬帳戶，請於 CSV 帳戶欄位明示」
- Q: CSV 含未定義的額外欄位時的處理策略？ → A: 靜默忽略 — 解析時以「第一行欄位名稱」做對應，本系統不認識的欄位直接略過不讀取，匯入照常進行；伺服端結構化日誌記錄被忽略的欄位名（含 user_id 與 action）以利 debug；前端 UI 不額外提示，避免不必要摩擦
- Q: 上限 20000 筆的長時間匯入 UX 模型？ → A: 同步阻塞 + 進度條 — UI 維持單頁 loading 狀態並顯示「已處理 X / N 筆」進度文字，後端每 500 筆向前端推送一次進度（SSE 或 short-poll 二擇一）；使用者離開頁面或網路中斷視為放棄，依 FR-014a 整批 rollback；不引入背景 job／通知系統的架構複雜度
- Q: 稽核日誌（`data_operation_audit_log`）的保留期限與清理策略？ → A: 管理員可調保留天數 — 預設 90 天；管理員於系統設定頁可改為 30／90／180／365 天或「永久」；系統每日午夜自動清理超過保留期的紀錄；管理員亦可於介面手動「匯出歷史紀錄為 CSV」並選擇清空
- Q: `backups/before-restore-{timestamp}.db` 自動備份檔的保留與清理策略？ → A: 保留最近 5 份 + 90 天上限（取較嚴格者）— 每次寫入新檔後系統自動刪除舊檔，超過 5 份或超過 90 天的一律清除；管理員介面顯示目前保留清單與檔案大小
- Q: 稽核日誌的查詢介面範圍？ → A: 管理員獨立查詢頁 + 一般使用者「我的操作紀錄」 — 管理員介面新增「稽核日誌」分頁（含 user_id／action／時間區間／result 過濾、時間倒序分頁、明細展開）；個人設定頁新增「我的操作紀錄」分頁（同樣支援過濾與分頁，但僅顯示自己的紀錄）
- Q: 使用者新增匯率時的「幣別代碼」驗證範圍？ → A: 內建 ISO 4217 白名單 — 系統內建 ISO 4217 主要流通幣別清單（約 180 個），UI 以下拉選擇或輸入後即時比對；非白名單值即拒絕並提示「不是有效的 ISO 4217 幣別代碼」；exchangerate-api.com 回 unsupported 仍維持 US5 Acceptance #5 的二次防線
- Q: 使用者「手動編輯匯率」時的數值合理性驗證範圍？ → A: 僅警告不拒絕 — 偏離即時匯率 ±20% 時 UI 顯示黃色警告與建議值（含目前 API 取得之即時值），但仍允許使用者「確認儲存」覆寫；空值、非數字、≤ 0 才硬性拒絕；無即時匯率比對基準時略過警告僅做 > 0 驗證

### User Story 1 - 把所有交易記錄一次匯出成 CSV 帶走（Priority: P1）

使用者進入「資料匯出」頁，點選「匯出交易記錄」，可選擇日期範圍（預設「全部」），按下「匯出」後系統下載一份 UTF-8 BOM 編碼的 CSV 檔案，欄位依序為：日期、類型、分類、金額、帳戶、備註。其中「分類」以「父分類 > 子分類」格式呈現（例如：「飲食 > 早餐」）。使用者用 Excel 或 Numbers 開啟後，中文不亂碼、欄位整齊、以 `=`／`+`／`-`／`@` 開頭的儲存格內容（如備註裡有「=SUM(A1)」）已被前置單引號轉義，不會在試算表中觸發公式執行。

**Why this priority**: 「能把資料帶走」是本功能存在的全部理由 — 使用者需要相信「即使網站關掉、即使我搬到別的工具，我的兩年記帳不會消失」。如果連匯出都做不到，整個系統的可信任度立刻歸零。匯出是匯入的前置（先有檔案才能再匯回去），屬最高優先 P1。

**Independent Test**: 一位有 500 筆交易（含支出、收入、轉出、轉入四種類型，跨多個帳戶與分類）的使用者按「匯出」；應立即得到一份 CSV：(1) 第一行為標題列（日期、類型、分類、金額、帳戶、備註），(2) 共 501 行（標題 + 500 筆），(3) 用 Excel 開啟中文顯示正常無亂碼，(4) 子分類欄位顯示「父 > 子」格式，無子分類者僅顯示父分類名稱，(5) 備註若含「=1+1」則匯出後該儲存格實際存「'=1+1」（單引號前綴），開啟後 Excel 顯示為文字而非公式結果，(6) 轉出與轉入兩筆對應交易各自獨立成行，類型欄位分別為「轉出」「轉入」。

**Acceptance Scenarios**:

1. **Given** 使用者於「資料匯出」頁不選日期範圍直接按匯出，**When** 點擊匯出按鈕，**Then** 下載含全部交易的 CSV，檔名為 `transactions-{YYYYMMDD}.csv`。
2. **Given** 使用者選擇日期範圍 2026-01-01 至 2026-03-31，**When** 點擊匯出，**Then** 僅匯出該區間的交易；區間外（如 2025-12-31 或 2026-04-01）的交易不會出現在 CSV。
3. **Given** 使用者某筆交易備註欄位內容為 `=SUM(1+1)`，**When** 匯出 CSV 並用 Excel 開啟，**Then** 該儲存格顯示為文字 `=SUM(1+1)`，不執行公式運算（內容已被前置單引號轉義）。
4. **Given** 使用者交易採用「飲食」父分類但無子分類，**When** 匯出，**Then** 該行分類欄位僅顯示「飲食」，不出現「>」符號。
5. **Given** 使用者於 2026-03-15 從帳戶 A 轉 1000 元到帳戶 B（系統建立兩筆 linked_id 關聯交易），**When** 匯出，**Then** CSV 中該日期出現兩行：一行類型為「轉出」、帳戶為 A；另一行類型為「轉入」、帳戶為 B；金額均為 1000。

### User Story 2 - 用 CSV 把交易記錄一次貼回系統（Priority: P1）

使用者於「資料匯入」頁拖曳或點擊上傳一份交易記錄 CSV（第一行為標題自動跳過）；上傳後系統先解析、顯示前 10 筆預覽（含類型、日期、分類、金額、帳戶、備註），讓使用者確認欄位對應正確；若 CSV 中出現使用者帳本中尚未建立的「分類」或「帳戶」名稱，系統彈出對話框列出全部缺項並詢問是否一鍵自動建立（同意則先建立分類／帳戶再匯入交易，拒絕則整批中止匯入）。匯入完成後顯示三個數字：成功 N 筆／略過 M 筆／錯誤 K 筆。「轉出」與「轉入」兩筆同日期＋同金額的紀錄會被系統自動兩兩配對並寫入相同 `linked_id`，重建轉帳關聯。

**Why this priority**: 沒有匯入，匯出就只是「資料的單行道」— 使用者無法從 Excel 整理好的歷史記錄一次貼回，也無法從本系統匯出再匯入做測試／驗證循環。轉帳自動配對是讓「匯入後能還原帳本完整邏輯」的關鍵。屬 P1，與 US1 構成本功能的核心 MVP。

**Independent Test**: 使用者上傳一份 1000 筆 CSV（含 50 對轉帳記錄，使用 3 個尚未建立的分類「醫療 > 牙醫」「投資 > 美股」「禮物」與 1 個未建立的帳戶「玉山銀行」）；系統行為應為：(1) 顯示前 10 筆預覽，(2) 彈窗列出 4 個缺項並詢問自動建立，(3) 使用者點「建立並匯入」後，系統先建分類（父在前子在後）與帳戶，再寫入 1000 筆交易，(4) 50 對轉出／轉入正確配對為 50 組共 100 筆 linked 記錄，(5) 完成後顯示「成功 1000／略過 0／錯誤 0」，(6) 使用者立刻於交易頁看到全部新增記錄。

**Acceptance Scenarios**:

1. **Given** 使用者上傳一份合法 CSV，**When** 系統解析完成，**Then** 顯示前 10 筆預覽 Modal，欄位包含日期／類型／分類／金額／帳戶／備註，並提示「共 N 筆，是否確認匯入」。
2. **Given** CSV 中出現尚未建立的分類「投資 > 美股」與帳戶「Firstrade」，**When** 使用者按下「確認匯入」，**Then** 系統彈出對話框：「偵測到 1 個新分類、1 個新帳戶，是否自動建立？」，使用者可選擇「建立並匯入」或「取消」。
3. **Given** CSV 中存在 1 筆 2026-03-15 / 1000 元 / 類型「轉出」/ 帳戶 A 與 1 筆同日期同金額類型「轉入」/ 帳戶 B，**When** 匯入完成，**Then** 兩筆交易被寫入相同 `linked_id`，於前端轉帳列表中可看到此配對。
4. **Given** 使用者上傳 28 MB 的 CSV，**When** 上傳，**Then** 系統拒絕並回應「檔案大小超過 25 MB 上限」；不開始解析。
5. **Given** 使用者上傳 22000 筆 CSV，**When** 系統解析完成，**Then** 拒絕匯入並提示「單次匯入上限 20000 筆，請拆分後再試」。
6. **Given** CSV 中有第 567 行金額欄位為「abc」（非數字），**When** 系統匯入，**Then** 該行歸類為「錯誤」並於完成後顯示「成功 999／略過 0／錯誤 1（第 567 行：金額格式無效）」；其餘合法行照常寫入。
7. **Given** CSV 中有 5 筆與既有交易完全相同（同日期、類型、分類、金額、帳戶、備註）的紀錄，**When** 匯入，**Then** 該 5 筆視為「略過」（避免重複寫入），完成後顯示「成功 N／略過 5／錯誤 0」。
8. **Given** CSV 類型欄位使用英文 `expense`／`income`／`transfer_out`／`transfer_in`，**When** 解析，**Then** 系統等同支援，與中文「支出／收入／轉出／轉入」對應一致。

### User Story 3 - 匯出與匯入「分類結構」CSV（Priority: P2）

使用者於資料匯出頁選「分類結構」，系統匯出 CSV，欄位為：類型、分類名稱、上層分類、顏色。父分類的「上層分類」欄位為空字串；子分類則填入父分類名稱。使用者可離線編輯（新增子分類、改顏色），再從匯入頁上傳；系統先建父後建子，已存在分類自動略過（依「類型 + 分類名稱」唯一鍵比對），完成後顯示「成功／略過／錯誤」筆數。

**Why this priority**: 分類結構是使用者的個人化資產，但每個帳本初期都要重建一次。提供 CSV 來回路徑讓「換新環境／在第二個帳本套用同一套分類樹」變得可能。屬 P2 — 沒有它仍可用 UI 一個一個建，但量大時極痛苦。

**Independent Test**: 使用者匯出含 8 個父分類與 25 個子分類（共 33 筆）的 CSV；於另一空白帳本上傳同一份檔案；應結果：(1) 33 筆全部成功寫入，(2) 父子關聯正確（子分類在 UI 樹狀結構顯示於對應父分類之下），(3) 顏色欄位完整保留，(4) 第二次重複匯入同檔，全部 33 筆顯示「略過」、「成功 0／錯誤 0」。

**Acceptance Scenarios**:

1. **Given** 使用者擁有「飲食」（無上層）與「飲食 > 早餐」「飲食 > 午餐」兩個子分類，**When** 匯出，**Then** CSV 三行 — 第一行「飲食」上層為空，第二、三行「早餐」「午餐」上層為「飲食」。
2. **Given** 上傳的 CSV 中子分類「早餐」的「上層分類」欄位填的「飲食」尚未建立，**When** 匯入，**Then** 系統先建立父分類「飲食」再建子分類「早餐」（CSV 內順序不影響結果）。
3. **Given** 使用者帳本已有支出分類「交通」，CSV 中亦存在類型＝支出／名稱＝「交通」，**When** 匯入，**Then** 該行視為「略過」，不覆寫既有顏色或 hierarchy。
4. **Given** 使用者擁有支出「交通」與收入「交通」（同名不同類型），**When** 匯出再匯入，**Then** 兩筆獨立保留，不會合併（唯一鍵 = 類型 + 名稱）。

### User Story 4 - 匯出與匯入股票交易、股利 CSV（Priority: P2）

使用者於匯出頁分別下載兩份獨立 CSV：

- **股票交易 CSV** 欄位：日期、股票代號、股票名稱、類型（買進／賣出，亦接受英文 buy／sell）、股數、成交價、手續費、交易稅、帳戶、備註。
- **股票股利 CSV** 欄位：日期、股票代號、股票名稱、現金股利、股票股利、帳戶、備註。

匯入時：(a) CSV 中的股票代號若使用者持倉中尚未建立，系統自動以代號 + 名稱新增該檔股票；(b) 若已存在但 `name` 欄位仍是代號（早期未取得 TWSE 名稱的舊資料），系統會以 CSV 中的名稱自動更新 — 也就是說「重新匯入一次最新的股票 CSV」就是修正錯誤名稱最省力的方式。匯入完成顯示成功／略過／錯誤筆數，並觸發 FIFO 全量重算（與 006 規格一致）。

**Why this priority**: 股票資料常需與券商月結單、CMoney／HiStock 等工具對齊，沒有 CSV 就無法做歷史補錄。但相對於「能否記帳」這個底座，股票匯入屬 P2。

**Independent Test**: 使用者匯出 100 筆股票交易 + 30 筆股利 CSV；其中 5 檔股票尚未建立持倉、3 檔股票既有名稱仍為代號（如「2330」而非「台積電」）；匯入相同 CSV：(1) 5 檔自動建立，(2) 3 檔名稱被更新為 CSV 中的正確名稱（如「台積電」），(3) 100 筆交易與 30 筆股利全部寫入，(4) FIFO 觸發重算後實現損益與 006 既有計算結果一致。

**Acceptance Scenarios**:

1. **Given** 使用者匯出股票交易，**When** 用 Excel 開啟，**Then** 欄位順序為：日期、股票代號、股票名稱、類型、股數、成交價、手續費、交易稅、帳戶、備註；類型欄位以中文「買進／賣出」呈現。
2. **Given** CSV 中類型欄位為英文 `buy`／`sell`，**When** 匯入，**Then** 系統等同視為「買進／賣出」處理。
3. **Given** CSV 含股票代號「6488」但持倉表中無此檔，**When** 匯入，**Then** 系統先建立持倉（symbol = 6488、name = CSV 中的名稱），再寫入該交易；FIFO 佇列同步初始化。
4. **Given** 持倉表中既有 symbol = 2330 / name = 「2330」（舊資料），CSV 中該股 name 欄為「台積電」，**When** 匯入，**Then** 系統 UPDATE 持倉 name 為「台積電」；既有交易紀錄不變。
5. **Given** CSV 中某行股利現金股利 = 500、股票股利 = 0、帳戶 = 「玉山銀行」、無備註，**When** 匯入，**Then** 寫入一筆現金股利紀錄並將 500 元入款至「玉山銀行」帳戶；不產生合成 $0 交易。
6. **Given** CSV 中某行現金股利 = 0、股票股利 = 100（股）、帳戶留空，**When** 匯入，**Then** 寫入股票股利紀錄並產生對應的合成 $0 買進交易（與 006 規格 FR 對齊）；FIFO 重算；帳戶欄位空白為合法（純股票股利不入款）。
7. **Given** CSV 中某行現金股利 = 500、帳戶欄位**空白**，**When** 匯入，**Then** 該行歸為「錯誤」，於完成 Modal 顯示原因「現金股利 > 0 時必填帳戶」；其他合法行不受影響。

### User Story 5 - 全球即時匯率自動更新（Priority: P2）

使用者於「匯率設定」頁可看到一份個人匯率清單（基礎貨幣固定 TWD），可自訂 3 碼幣別代碼（如 USD、JPY、EUR）加入清單；頁面提供開關「自動更新」與按鈕「立即取得即時匯率」，按下後系統呼叫 exchangerate-api.com 同步當前匯率並顯示「上次取得時間」精確到秒（例如「2026-04-27 13:25:08」）。系統內部以 30 分鐘為單位跨使用者共享匯率快取（若 5 分鐘前 A 使用者已取得 USD/TWD = 32.45，B 使用者於 4 分鐘後查詢時直接命中快取，不再呼叫外部 API）；每筆匯率有 `is_manual` 欄位區分手動／自動，自動更新只覆蓋 is_manual = false 的紀錄，不動使用者手動鎖定的匯率。免費版 API 不需 API key、付費版透過 `EXCHANGE_RATE_API_KEY` 環境變數啟用。

**Why this priority**: 多幣別記帳是進階使用情境（出國旅遊、海外薪資、海外股票股利），手動維護匯率非常痛苦但「不更新就會看到錯誤的台幣換算」。對使用此功能的人而言極關鍵；對僅用 TWD 的使用者而言可有可無。屬 P2。

**Independent Test**: 一位使用者於匯率設定頁加入 USD 與 JPY，按「立即取得即時匯率」；應結果：(1) USD 與 JPY 的 rate 欄位被填入（例：USD=32.45、JPY=0.21），(2) 上次取得時間精確到秒顯示，(3) 30 分鐘內第二次按按鈕不應對外發 request（命中快取，UI 顯示「使用 N 分鐘前的快取」），(4) 使用者手動將 USD 改為 33.00 並標記為手動鎖定後，再按自動更新時 USD 不被覆寫，僅 JPY 被更新。

**Acceptance Scenarios**:

1. **Given** 使用者尚未設定匯率，**When** 進入匯率設定頁，**Then** 看到基礎貨幣 TWD 固定（不可刪除）與「+ 新增幣別」按鈕。
2. **Given** 使用者輸入「USD」並按新增，**When** 自動更新開啟，**Then** 系統呼叫 exchangerate-api.com 取得 USD/TWD 匯率並寫入清單，is_manual = false。
3. **Given** 使用者點按「立即取得即時匯率」且快取時間 ≤ 30 分鐘，**When** 系統處理，**Then** 不發 HTTP request，直接從快取讀取，UI 顯示快取狀態。
4. **Given** 系統呼叫 API 失敗（網路錯誤或 API 服務 down），**When** 使用者請求更新，**Then** UI 顯示明確錯誤訊息（例：「即時匯率服務暫時無法連線，請稍後再試」），既有匯率不被清空。
5. **Given** 使用者輸入未支援之幣別代碼「XYZ」，**When** 按新增，**Then** API 回傳 unsupported，系統提示「幣別 XYZ 不被即時匯率服務支援」，不寫入清單。
6. **Given** 使用者手動編輯 USD 匯率為 33.00 並按下「鎖定為手動匯率」，**When** 自動更新觸發，**Then** USD 不被覆寫，is_manual = true 持續保留。

### User Story 6 - 管理員整檔備份／還原資料庫（Priority: P3）

具管理員權限的使用者於系統設定頁看到「資料庫備份」區塊，可按下「下載備份」匯出整個未加密的 SQLite 檔案（檔名 `assetpilot-backup-{YYYYMMDDHHmmss}.db`）；還原時上傳一份 .db 檔，系統先驗證檔案格式（SQLite header）與必要資料表結構（schema 必須包含 transactions、accounts、categories、stocks 等核心表），驗證通過後**自動備份目前現行資料庫**到 `backups/before-restore-{timestamp}.db`，再覆寫資料庫並提示重新登入。一般使用者於 UI 看不到此功能；備份檔案路徑必須排除於 `.gitignore` 與 `.dockerignore`，避免誤入 commit 或 image。

**Why this priority**: 災難復原是「最後一道防線」— 一般使用者不會用，但維運者一旦需要就是救命操作。屬 P3，因為一般使用者完全不會碰到，產品力上 US1 / US2 才是讓使用者覺得「資料安全」的功能。

**Independent Test**: 管理員下載一份 SQLite 備份檔（5 MB）；於另一台空白部署環境登入管理員帳號上傳該檔；驗證：(1) 系統正確識別為合法 SQLite 並通過 schema 驗證，(2) 還原前自動於 `backups/` 目錄生成 `before-restore-{timestamp}.db`，(3) 還原成功後重新登入，看到完整使用者帳號與全部交易記錄，(4) 一般使用者登入後設定頁完全看不到「資料庫備份」區塊。

**Acceptance Scenarios**:

1. **Given** 一般使用者登入並進入設定頁，**When** 觀察畫面，**Then** 不出現「資料庫備份」區塊；對應 API 端點若被直接呼叫則回應 403。
2. **Given** 管理員按下「下載備份」按鈕，**When** 系統處理，**Then** 先彈出確認 Modal 警示「此檔案包含明文敏感資料（含使用者密碼雜湊、session token 等），請妥善保管於受控環境，勿透過 email／雲端硬碟轉存」；管理員點「確認下載」才觸發下載，點「取消」則不下載。
3. **Given** 管理員上傳一個 .txt 檔（非 SQLite），**When** 系統驗證，**Then** 拒絕並回應「檔案格式不符，必須為 SQLite 資料庫」。
4. **Given** 管理員上傳一個合法 SQLite 但缺少 `transactions` 資料表，**When** 系統驗證，**Then** 拒絕並列出缺少的資料表清單。
5. **Given** 管理員上傳合法備份檔且驗證通過，**When** 系統執行還原，**Then** 先寫 `backups/before-restore-{timestamp}.db`、再以新檔覆寫資料庫；UI 顯示「還原完成，請重新登入」。
6. **Given** 系統工作目錄含已產生的 `.db` 備份檔，**When** 開發者執行 `git status` 或 `docker build`，**Then** 該檔不被列入 staged／不被 COPY 進 image（已加入 `.gitignore` 與 `.dockerignore`）。

### Edge Cases

- **空檔案上傳**：使用者上傳 0 byte 或僅含標題列的 CSV → 系統明示「檔案中無資料列」，不視為錯誤但也不寫入。
- **檔案大小超限（> 25 MB）**：拒絕並提示限制值，不開始解析。
- **筆數超限（> 20000 列）**：解析後立即拒絕，提示「單次匯入上限 20000 筆，請拆分後再試」。
- **編碼非 UTF-8**：嘗試以 UTF-8 BOM、UTF-8 無 BOM、Big5 探測；探測失敗時提示使用者另存為 UTF-8。
- **欄位名稱被使用者改成英文／順序不同**：系統以**第一行的欄位名稱**做欄位對應而非依靠位置；找不到必要欄位（如「日期」「金額」）時拒絕並列出缺欄位名。
- **CSV 含本系統未定義的額外欄位**：解析時靜默忽略不讀取，匯入照常進行；伺服端結構化日誌記錄被忽略的欄位名供事後 debug；UI 不額外提示。
- **Formula Injection 字元出現在備註／分類名稱**：匯出時自動前置單引號轉義（`=`、`+`、`-`、`@` 開頭）；匯入時如使用者**未**清除前置單引號則照原文存入（單引號不被視為公式語法字元）。
- **轉帳配對失敗**：CSV 中存在「轉出 1000 / 2026-03-15」但找不到對應「轉入 1000 / 2026-03-15」（或對方筆數不對等）→ 該筆視為單向轉出寫入並記為「警告」（非錯誤），於完成 Modal 列出未配對清單供人工後處理。
- **重複匯入同一檔**：以「日期＋類型＋分類＋金額＋帳戶＋備註」六欄完全相同視為重複，自動略過，避免變成「按一次新增 1000 筆」。
- **股票代號 case 敏感性**：CSV 中代號 `2330` 與 `2330 ` (含空白) 視為同一檔；自動 trim。
- **股利 CSV 同一日期同一檔多筆紀錄**：允許（例：除息與發放日不同的特殊狀況）；不視為重複。
- **匯率服務 API 限額耗盡**：API 回傳 quota exceeded → UI 提示「即時匯率配額已用盡，請於下個重置週期再試或升級為付費版」。
- **管理員備份還原時資料庫正在被寫入**：系統於還原前進入 read-only 狀態（拒絕新 mutation 請求 30 秒），完成後解除；若有衝突的進行中操作，於還原完成 UI 提示使用者該操作可能未存。
- **還原覆寫主資料庫過程中失敗**：系統 MUST 自動以還原前已寫入的 `backups/before-restore-{timestamp}.db` 回滾主資料庫；UI 提示「還原失敗，已自動回復至還原前狀態」並要求管理員檢查 server 日誌（FR-026a）。
- **同使用者並行匯入**：兩個分頁各自上傳 CSV 時，後到的請求 MUST 回應 HTTP 409 `IMPORT_IN_PROGRESS`；UI 提示「您已有匯入進行中」並禁用上傳按鈕；待前一個匯入完成或失敗 rollback 後使用者可重試。

## Requirements *(mandatory)*

### Functional Requirements

#### A. 交易記錄匯出／匯入

- **FR-001**: 系統 MUST 提供「匯出交易 CSV」功能，產出 UTF-8 BOM 編碼檔案，欄位順序固定為：日期、類型、分類、金額、帳戶、備註。
- **FR-002**: 匯出 MUST 支援使用者選擇日期範圍（起／訖日期），未選擇時匯出全部。
- **FR-003**: 「分類」欄位 MUST 以「父分類 > 子分類」格式呈現；無子分類時僅輸出父分類名稱。
- **FR-004**: 「類型」欄位 MUST 支援四值：支出、收入、轉出、轉入；匯出以中文表示。
- **FR-005**: 系統 MUST 對所有匯出的儲存格內容套用 Formula Injection 防護：以 `=`、`+`、`-`、`@` 開頭的值前置單引號（`'`）。
- **FR-006**: 系統 MUST 提供「匯入交易 CSV」端點與前端拖曳／點擊上傳 UI；接收 CSV 後跳過第一行（標題）。
- **FR-006a**: 所有 CSV 匯入端點（交易、分類、股票交易、股利）MUST 以「第一行欄位名稱」做欄位對應；遇到本系統未定義的額外欄位時 MUST 靜默忽略（不讀取、不警告 UI），匯入照常進行；伺服端結構化日誌 MUST 紀錄被忽略的欄位名稱（含 user_id、action、檔名）以利後續 debug。必要欄位缺失（如交易 CSV 缺「日期」或「金額」）的拒絕行為仍依 Edge Cases 既有規則處理。
- **FR-007**: 系統 MUST 在匯入前顯示前 10 筆預覽，包含類型、日期、分類、金額、帳戶、備註欄位。
- **FR-008**: 匯入 CSV 端點 MUST 以 25 MB 與 20000 筆為上限，超過任一上限即拒絕並回應原因。
- **FR-009**: 「類型」欄位匯入 MUST 同時接受中文（支出／收入／轉出／轉入）與英文（expense／income／transfer_out／transfer_in），大小寫不敏感。
- **FR-010**: 「分類」欄位匯入 MUST 支援「父分類 > 子分類」格式；解析時依該格式找對應分類。
- **FR-011**: 匯入時若偵測到 CSV 含未建立的分類或帳戶，系統 MUST 集中列出全部缺項並彈窗詢問是否一鍵自動建立；使用者選擇取消則整批中止匯入（不部分寫入）。
- **FR-012**: 系統 MUST 將同日期、同金額的「轉出」與「轉入」紀錄自動兩兩配對並寫入相同 `linked_id`，重建轉帳關聯。配對演算法（多重候選消歧）：(a) 先依「日期 + 金額」分組；(b) 同組內若兩邊都還有未配對候選，依 CSV 中出現順序逐一兩兩配對（第 N 筆轉出 ↔ 第 N 筆轉入）；(c) 多出的單側候選（一邊全配完、另一邊還有剩餘）寫為單向交易並列入「警告」清單供人工後處理。
- **FR-013**: 匯入完成 MUST 顯示三個彙總數字：成功 N／略過 M／錯誤 K；錯誤項 MUST 附原 CSV 行號與失敗原因（最多顯示前 50 條）。
- **FR-014**: 系統 MUST 將「日期＋類型＋分類＋金額＋帳戶＋備註」六欄完全相同的匯入紀錄視為重複，自動略過（不建立新記錄）。
- **FR-014a**: 「自動建立缺項 + 匯入交易」MUST 為單一原子操作（包覆於同一 DB transaction）— 自動建立的分類／帳戶與所有交易紀錄 commit 同生共死；任一階段失敗（包括建立缺項階段、交易寫入階段、轉帳配對階段）即整批 rollback，使用者重試時環境為乾淨白板。失敗時 UI MUST 顯示明確錯誤訊息與失敗位置（CSV 行號或階段名稱）。
- **FR-014b**: 所有 CSV（交易、分類、股票交易、股利）中「日期」欄位 MUST 嚴格採 ISO 8601 格式 `YYYY-MM-DD`；匯出時固定輸出此格式，匯入時僅接受此格式，其他格式（含 `YYYY/MM/DD`、`MM/DD/YYYY`、`民國 YYY/MM/DD` 等）一律視為「錯誤」並附原因「日期格式必須為 YYYY-MM-DD」。
- **FR-014c**: 系統 MUST 對所有匯入端點（交易、分類、股票交易、股利）施加「同使用者單一匯入排他鎖」— 同一 `user_id` 同時只允許 1 個匯入請求進行中；第二個並行請求 MUST 回應 HTTP 409 Conflict，body 含原因碼 `IMPORT_IN_PROGRESS`；前端 MUST 同時依本地狀態禁用上傳按鈕，並於彈窗提示「您已有匯入進行中，請稍候完成後再試」。鎖 MUST 在匯入正常完成、失敗 rollback、或 server 重啟時自動釋放（不留殘留鎖）。
- **FR-014d**: 所有 CSV 匯入端點 MUST 採同步阻塞模型，並支援進度回饋：後端 MUST 在匯入過程中每處理 500 筆（最後一批不足 500 亦推送）對前端推送一次進度事件（採 Server-Sent Events 或前端 short polling 任一機制），event payload 至少包含 `processed`、`total`、`phase`（`parsing`／`validating`／`writing`／`pairing`／`finalizing`）。前端 UI MUST 顯示「已處理 X / N 筆」進度文字與對應 phase 標籤；使用者離開頁面或網路中斷視為放棄，依 FR-014a 整批 rollback。本功能 MUST NOT 採用背景 job 或站內通知架構。

#### B. 分類結構匯出／匯入

- **FR-015**: 系統 MUST 提供分類結構匯出，欄位為：類型、分類名稱、上層分類、顏色；父分類「上層分類」為空字串、子分類填父分類名稱。**顏色欄位格式**：MUST 嚴格採 hex 6 碼 `#RRGGBB`（大小寫不限）；匯出固定輸出此格式，匯入時以正規式 `^#[0-9A-Fa-f]{6}$` 驗證，3 碼短寫（`#F53`）、CSS 顏色名（`red`）、rgb() 函式等格式一律拒絕並歸為「錯誤」附原因「顏色格式必須為 #RRGGBB」。
- **FR-016**: 匯入時系統 MUST 先建父分類後建子分類（不論 CSV 內順序），確保 hierarchy 完整。
- **FR-017**: 系統 MUST 以「類型 + 分類名稱」為唯一鍵；既有分類於匯入時自動略過，不覆寫顏色或 hierarchy。

#### C. 股票交易／股利匯出／匯入

- **FR-018**: 系統 MUST 提供「股票交易 CSV」匯出／匯入，欄位為：日期、股票代號、股票名稱、類型、股數、成交價、手續費、交易稅、帳戶、備註；類型匯出採中文（買進／賣出），匯入接受中英文。
- **FR-019**: 系統 MUST 提供「股票股利 CSV」匯出／匯入，欄位為：日期、股票代號、股票名稱、現金股利、股票股利、帳戶、備註。**帳戶欄位語意**：現金股利 > 0 時 MUST 為現金入款帳戶名稱（必填）；純股票股利（現金股利 = 0）時帳戶可留空（與 006 規格 Key Entity「股利」對齊）。匯入時若現金股利 > 0 而帳戶欄位空白或對應帳戶不存在則該行歸類為「錯誤」並附原因。
- **FR-020**: 股票 CSV 匯入時，若代號未建立持倉，系統 MUST 自動依代號 + 名稱新增該檔股票並初始化 FIFO 佇列。
- **FR-021**: 股票 CSV 匯入時，若代號已存在但持倉的 `name` 欄位仍為代號（與 `symbol` 相同），系統 MUST 以 CSV 中的名稱自動 UPDATE；既有交易紀錄不受影響。
- **FR-022**: 股票交易匯入 MUST 觸發 FIFO 全量重算（與 006 規格 FR-018 對齊）。
- **FR-023**: 股利匯入時，若「股票股利」> 0，系統 MUST 同步建立合成 $0 買進交易（與 006 規格對齊）。
- **FR-023b**: 純股票股利（現金股利 = 0、股票股利 > 0、CSV 帳戶欄位留空）匯入時，FR-023 所建立之合成 $0 買進交易的帳戶判定 MUST 依下列順序解析：(1) 查詢該 `symbol` 既有買進交易中時間最近的一筆，取其 `account_id`；(2) 若該股票無任何既有買進紀錄（首次匯入），且使用者僅存在一個證券型帳戶（type 屬證券／投資類），則使用該唯一帳戶；(3) 若使用者有多個證券帳戶且該股票無既有歷史，該 CSV 行 MUST 歸為「錯誤」並附原因「純股票股利合成交易無法判定所屬帳戶，請於 CSV 帳戶欄位明示」；其他合法行不受影響。若 CSV 帳戶欄位有填值（即使現金股利 = 0），MUST 直接以該帳戶為合成交易的綁定帳戶（顯式優先於推導）。
- **FR-023a**: 股票交易匯入 MUST 將「日期 + 代號 + 類型 + 股數 + 成交價 + 帳戶」六欄完全相同的紀錄視為重複，自動略過；股利匯入 MUST 將「日期 + 代號 + 現金股利 + 股票股利」四欄完全相同的紀錄視為重複，自動略過。重複偵測在寫入前先行；略過項計入「略過 M」彙總並適用 SC-009 冪等性保證。

#### D. 資料庫整檔備份／還原（管理員）

- **FR-024**: 系統 MUST 提供管理員專屬端點，匯出整個未加密 SQLite 資料庫檔案；檔名格式 `assetpilot-backup-{YYYYMMDDHHmmss}.db`。
- **FR-024a**: 管理員按下「下載備份」時，系統 MUST 先顯示一次性確認彈窗，明示：「此檔案包含明文敏感資料（含使用者密碼雜湊、session token、授權憑證等），請妥善保管於受控環境，勿透過 email／公開雲端硬碟轉存。」管理員點按「確認下載」後才觸發實際下載；點「取消」則不下載。系統 MUST NOT 自行加密或以密碼包裝備份檔（避免密碼遺失即無法還原的災難情境）。
- **FR-025**: 還原端點 MUST 先驗證上傳檔案的 SQLite header 與必要資料表存在性（transactions、accounts、categories、stocks 等核心表），任一驗證失敗即拒絕。
- **FR-026**: 還原成功覆寫前，系統 MUST 自動將現行資料庫複製到 `backups/before-restore-{timestamp}.db`。
- **FR-026a**: 還原過程中若覆寫主資料庫的步驟失敗（disk full、process killed、I/O error 等任何原因導致主庫處於毀損／半寫入狀態），系統 MUST 自動以 FR-026 所建立的 `backups/before-restore-{timestamp}.db` 覆蓋回主資料庫，使資料回復至還原操作前的狀態；UI MUST 顯示「還原失敗，已自動回復至還原前狀態，請檢查 server 日誌」。失敗事件 MUST 寫入 server 日誌（含失敗原因、時間戳、回滾使用的備份檔案路徑），供維運人員後續排查。若自動回滾本身亦失敗（極端情境，如備份檔同時遺失），系統 MUST 於日誌與 UI 同時明示主庫狀態未知並列出可用備份檔路徑供管理員手動處置。
- **FR-027**: 一般使用者 MUST 不可於 UI 看到備份／還原入口；對應端點 MUST 由 RBAC 限制為僅管理員可呼叫，其他角色一律 403。
- **FR-028**: 系統 MUST 將備份檔案與其資料夾路徑加入 `.gitignore` 與 `.dockerignore`，避免進入版本控制或映像檔。
- **FR-026b**: 系統 MUST 對 `backups/before-restore-{timestamp}.db` 自動備份檔施加保留上限：每次寫入新檔後，自動刪除既有檔案中超過「最近 5 份」或「超過 90 天」（兩條件取較嚴格者）的舊檔；刪除作業 MUST 寫入 server 結構化日誌。管理員介面 MUST 列出目前所有 `before-restore` 檔的時間戳、檔案大小與佔用總空間；管理員 MUST 可於介面手動刪除指定檔案（需二次確認）。 `assetpilot-backup-{YYYYMMDDHHmmss}.db`（管理員主動下載的備份檔）不在此自動清理範圍內，僅由管理員自行管理。

#### E. 全球即時匯率自動更新

- **FR-029**: 系統 MUST 提供匯率設定頁，顯示使用者個人匯率清單；基礎貨幣固定 TWD（不可刪除、不可修改）。
- **FR-030**: 使用者 MUST 可自訂 3 碼幣別代碼（如 USD、JPY）加入個人清單；系統 MUST 內建 ISO 4217 主要流通幣別白名單（約 180 個代碼）並依此白名單驗證輸入：UI 提供下拉選擇或輸入後即時比對；非白名單值 MUST 立即拒絕並提示「不是有效的 ISO 4217 幣別代碼」，不寫入清單。白名單以靜態資料檔（JSON 或設定檔）維護，新增／移除代碼不需重新部署應用。即使通過 ISO 4217 驗證，仍可能於呼叫 exchangerate-api.com 時收到 unsupported 回應（US5 Acceptance #5 的二次防線）— 該情境照原有錯誤處理。
- **FR-031**: 系統 MUST 提供「自動更新」開關與「立即取得即時匯率」按鈕；按下按鈕即同步當前匯率並更新「上次取得時間」（精確到秒）。
- **FR-032**: 系統 MUST 維護跨使用者共享的匯率快取（30 分鐘 TTL）；快取命中時不發 HTTP request 給 exchangerate-api.com。
- **FR-033**: 每筆匯率紀錄 MUST 含 `is_manual` 欄位區分手動／自動；自動更新只覆寫 `is_manual = false` 的紀錄。
- **FR-033a**: 使用者手動編輯匯率時，系統 MUST 採「警告但不拒絕」的數值驗證模型：(1) 空值、非數字、`≤ 0` 的輸入 MUST 硬性拒絕並提示原因；(2) 通過 (1) 後，若該幣別存在系統最近一次自動取得的即時匯率（`is_manual = false` 紀錄或快取項），且使用者輸入值偏離該基準 ±20% 以上，UI MUST 顯示黃色警告含「目前即時匯率為 X.XXXX，您輸入 Y.YYYY 偏離 Z%，請確認」，但提供「確認儲存」按鈕允許覆寫；(3) 若該幣別無任何即時匯率比對基準（首次新增且 API 未成功），略過 ±20% 警告，僅執行 (1) 的硬性檢查。
- **FR-034**: 系統 MUST 同時支援免費版（無需 API key）與付費版（透過 `EXCHANGE_RATE_API_KEY` 環境變數）；環境變數存在時優先使用付費端點。
- **FR-035**: 匯率服務不可用時，UI MUST 顯示明確錯誤訊息且不清空既有匯率值。

#### F. API 使用與授權資訊頁

- **FR-036**: 系統 MUST 於側邊欄提供獨立頁面，集中列出所有對外 API、用途、來源連結與授權注意事項。
- **FR-037**: IPinfo 條目 MUST 顯示 `IP address data is powered by IPinfo` 授權字樣（合規要求）。
- **FR-038**: 條目資料 MUST 至少涵蓋：exchangerate-api.com、IPinfo、TWSE（openapi.twse.com.tw 與 www.twse.com.tw）、Google Identity Services、Resend；未來新增第三方服務時可快速擴充。

#### G. 範圍限制（明示不做）

- **FR-039**: 系統 MUST NOT 提供 Excel (.xlsx) 匯出／匯入；僅提供 CSV。
- **FR-040**: 系統 MUST NOT 支援 OFX、QIF 等銀行對帳格式的匯入。
- **FR-041**: 系統 MUST NOT 內建自動雲端同步（Google Drive、Dropbox 等）；備份／還原為手動觸發的下載／上傳模型。

#### H. 資料操作稽核日誌（Data Operation Audit Log）

- **FR-042**: 系統 MUST 維護一張統一稽核表 `data_operation_audit_log`，紀錄所有匯出／匯入／備份／還原操作（含一般使用者與管理員）。表欄位至少包含：`id`、`user_id`、`role`、`action`、`ip_address`、`user_agent`、`timestamp`（UTC ISO 8601）、`result`（`success` / `failed` / `rolled_back`）、`is_admin_operation`（boolean）、`metadata`（JSON：檔案大小、筆數、失敗原因、相關備份檔路徑等）。
- **FR-043**: `action` 列舉值 MUST 至少涵蓋：`export_transactions`、`import_transactions`、`export_categories`、`import_categories`、`export_stock_transactions`、`import_stock_transactions`、`export_stock_dividends`、`import_stock_dividends`、`download_backup`、`restore_backup`、`restore_failed`。
- **FR-044**: 稽核日誌 MUST 於對應操作完成（成功）、失敗 rollback、或自動回滾後立即寫入；寫入失敗 MUST NOT 阻斷主操作的完成回應，但 MUST 寫入 server 結構化日誌以利事後補登。
- **FR-045**: 管理員 MUST 可於管理員介面查詢稽核日誌（依 user_id、action、時間區間過濾）；一般使用者 MUST 僅能查詢自己的稽核紀錄；對應端點 MUST 由 RBAC 限制。
- **FR-046**: 稽核日誌 MUST NOT 包含明文敏感資料（不記錄 CSV 內容、不記錄密碼、不記錄 session token）；僅紀錄行為元資料。
- **FR-046a**: 系統 MUST 提供稽核日誌保留期間設定，預設值 = 90 天；管理員 MUST 可於系統設定頁將保留天數調整為下列其中之一：30、90、180、365 或「永久」（不清理）。系統 MUST 於每日午夜（伺服器時區）執行清理 job，刪除 `timestamp` 早於「現在 - 保留天數」的紀錄；保留設定為「永久」時跳過清理。管理員 MUST 可於同一頁面手動匯出當前所有稽核紀錄為 CSV（檔名 `audit-log-{YYYYMMDDHHmmss}.csv`），並可於匯出後選擇「清空全部」（需二次確認）。
- **FR-046b**: 管理員介面 MUST 提供獨立的「稽核日誌」分頁，支援以下過濾條件：`user_id`、`action`（多選）、時間區間（start／end datetime）、`result`（success／failed／rolled_back）；列表 MUST 以 `timestamp` 倒序分頁顯示（每頁 50 筆）；每筆 MUST 可展開查看 `metadata` JSON 詳情；該頁面 MUST 由 RBAC 限制為僅管理員可進入。一般使用者個人設定頁 MUST 新增「我的操作紀錄」分頁，提供相同過濾與分頁能力，但僅顯示其自身的紀錄；對應 API endpoint MUST 強制以登入使用者 ID 過濾，禁止跨使用者查詢。

### Key Entities *(include if feature involves data)*

- **CSV 匯出檔（Transactions Export）**：UTF-8 BOM；欄位 [日期、類型、分類、金額、帳戶、備註]；經 Formula Injection 防護處理。
- **CSV 匯出檔（Categories Export）**：欄位 [類型、分類名稱、上層分類、顏色]；父子順序由匯入端決定。
- **CSV 匯出檔（Stock Transactions Export）**：欄位 [日期、股票代號、股票名稱、類型、股數、成交價、手續費、交易稅、帳戶、備註]。
- **CSV 匯出檔（Stock Dividends Export）**：欄位 [日期、股票代號、股票名稱、現金股利、股票股利、帳戶、備註]；現金股利 > 0 時帳戶必填。
- **匯入結果摘要（ImportSummary）**：{successful: N, skipped: M, errors: [{row, reason}], warnings: [{row, reason}]}；於匯入完成 Modal 顯示。
- **匯率快取項（ExchangeRateCacheEntry）**：{currencyCode, rate, fetchedAt, source: "free|paid"}；TTL 30 分鐘，跨使用者共享。
- **使用者匯率紀錄（UserExchangeRate）**：{userId, currencyCode, rate, isManual, lastUpdatedAt}；is_manual 為 true 時不被自動更新覆寫。
- **資料庫備份檔（DatabaseBackupFile）**：未加密 SQLite 二進位檔；檔名 `assetpilot-backup-{timestamp}.db`；僅管理員可下載／上傳。
- **API 條目（ExternalApiEntry）**：{name, description, url, attribution?}；於 API 資訊頁集中列出。
- **資料操作稽核紀錄（DataOperationAuditLog）**：{id, userId, role, action, ipAddress, userAgent, timestamp（UTC ISO 8601）, result（success｜failed｜rolled_back）, isAdminOperation, metadata（JSON：檔案大小、筆數、失敗原因、備份檔路徑等元資料；MUST NOT 含明文敏感資料）}；涵蓋所有匯出／匯入／備份／還原操作。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者匯出一年量交易（約 5,000 筆）的 CSV 檔，從點擊匯出到下載開始 ≤ 5 秒（90 百分位）。
- **SC-002**: 使用者匯入 1,000 筆交易（含 100 對轉帳）的 CSV，從上傳完成到 UI 顯示「成功 N／略過 M／錯誤 K」 ≤ 30 秒（90 百分位）；匯入進行期間 UI MUST 至少每 500 筆更新一次「已處理 X / N 筆」進度文字（FR-014d）。
- **SC-003**: 100% 的匯出 CSV 在 Excel／Numbers 開啟時，含 `=`／`+`／`-`／`@` 起始字元的儲存格不觸發公式執行（驗證以 OWASP Formula Injection cheatsheet 範例樣本檢測）。
- **SC-004**: 轉帳記錄經「匯出 → 清空 → 匯入」往返後，自動配對成功率 ≥ 99%（同日期同金額且兩端帳戶皆存在的 case）。
- **SC-005**: 即時匯率「立即取得即時匯率」按鈕成功更新使用者清單的成功率 ≥ 95%（剩餘 5% 為 API 服務本身不穩定）；快取命中（30 分鐘內第二次請求）時間 ≤ 100 ms。
- **SC-006**: 管理員備份／還原一份 50 MB 的 SQLite 資料庫，從上傳完成到 UI 提示「還原完成」≤ 60 秒；還原前自動產生的 `backups/before-restore-*.db` 100% 存在且可被再次還原。
- **SC-007**: 使用者首次使用匯入功能時，能在 5 分鐘內完成（從進入匯入頁到看到「成功 N」結果，含上傳一份範例 CSV、解決一個缺項建立提示）— 衡量教學門檻是否合理。
- **SC-008**: API 使用與授權資訊頁覆蓋本系統 100% 對外呼叫的第三方 API（以實際 codebase 中出現的 outbound HTTP client 清單比對）；IPinfo 授權字樣 100% 出現。
- **SC-009**: 重複匯入同一份 CSV 時，「略過」筆數 = 第一次匯入的「成功」筆數，「成功」= 0；保證匯入操作具冪等性。

## Assumptions

- 使用者主要透過 Excel／Numbers／Google Sheets 編輯 CSV；UTF-8 BOM 為 Excel 中文不亂碼的事實標準。
- 25 MB／20000 筆上限源於 body size 限制；對家庭記帳場景（每年 ~5000 筆）已綽綽有餘，超出此量的使用者會主動拆分檔案。
- 「分類」與「帳戶」名稱於同型別下具唯一性（不重名）；本規格依此前提設計缺項自動建立邏輯。
- 股票交易匯入觸發 FIFO 全量重算與既有 006 規格 FR-018 對齊；無需另開「匯入時是否重算」選項。
- 匯率服務 exchangerate-api.com 的 30 分鐘快取為跨使用者共享；本系統基礎貨幣固定 TWD，不需處理多基礎貨幣議題。
- 管理員角色判定沿用既有 001-user-permissions 的 RBAC；本規格不重新定義權限模型。
- 「上次取得時間」儲存為 UTC ISO 8601；UI 顯示時依使用者時區轉換並精確到秒。
- API 使用與授權頁的條目以資料來源（例如固定 JSON 或設定檔）驅動，非硬編碼於前端元件 — 新增條目不需修改多個檔案即可上架。
- 資料庫備份還原是「目的端整顆覆蓋來源端」的模型；不支援部分 merge 或欄位級的 conflict resolution。
- 匯出檔名含時間戳（`YYYYMMDD` 或 `YYYYMMDDHHmmss`），避免使用者連續匯出多份時瀏覽器自動加 ` (1)`／` (2)` 後綴。
